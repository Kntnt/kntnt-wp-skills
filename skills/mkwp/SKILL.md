# mkwp

Create a fresh local WordPress site by driving the `mkwp` command. `mkwp` is **not** part of the shared transfer engine `clone` and `pull` run — there is no production site, no control channel, no baseline, nothing to import. It starts from nothing: gather the flags `mkwp` itself needs, confirm the ones context cannot settle, and run `mkwp` once. `--yes` accepts every recommendation, never pauses, and prints a full decided-and-done record.

Read `docs/spec.md` (the `mkwp` section) and `CONTEXT.md` (the glossary) alongside this file. Where a literal here and the spec diverge, the spec wins.

## 0. Help gate

If the arguments are `help`, `--help`, or `-h`, run `uv run "${CLAUDE_PLUGIN_ROOT}/scripts/help.py" mkwp`, emit its output verbatim as Markdown, and stop. Do nothing else.

## 1. Dependency check

Mandatory before scaffolding, running the **local** portion of `clone`/`pull`'s own dependency step (its §1) — the same `ddev version` plus Docker/Colima liveness probe and required-CLI-tool roster, so the two skills never silently define "local dependencies" two different ways. Production-side checks (the Kntnt Extractor status/environment handshake, the capability probe, the download preflight) do not apply here — `mkwp` never touches production, there is nothing to check on that side. On any failure, abort immediately: report what to install or fix, where to get it, and the exact command to re-run once done. The agent never installs system software itself; where a fix is safely agent-runnable (e.g. `brew install jq`), offer to run it as its own accept-or-override gate before re-checking, rather than assuming consent. This gate is never auto-accepted under `--yes` — an unattended run has no operator present to consent to installing system software, so `--yes` aborts with the remediation message instead of running the fix.

1. **`ddev` and its container backend.** `ddev` is on `PATH` and its container backend actually responds — `ddev version` exits zero, and a cheap Docker/Colima liveness probe (`docker info`, else `colima status`) also succeeds; a `ddev` binary with a stopped backend otherwise fails silently deep inside the scaffold step's (§4) first `ddev start`.
2. **`mkwp` itself.** Run `mkwp --help` locally. If the command is not found on `PATH` at all, that is itself the failure — do not fabricate help output. Pipe the result to `uv run "${CLAUDE_PLUGIN_ROOT}/scripts/mkwp_guard.py"` as `{ "helpOutput": <the captured stdout, or null when mkwp was not found> }`. This is the **one place** the guard's pass/fail verdict and remediation are computed — `clone`'s own equivalent step (§1 of its health check, the dependency step) reads the same helper rather than re-deriving the check. If the verdict's `ok` is `false`, abort immediately and print `reason` and `remediation` verbatim — no gate, no further step, no fabricated stack trace. The operator installs or upgrades binaries; this skill does not.
3. **Every other required CLI tool.** On `PATH`: `uv`, `jq`, and `curl` — the same baseline `clone`/`pull`'s own dependency step checks. (No separate checksum or encryption binary is needed: the sealed-container unseal `clone`/`pull` run is a `uv`-run helper with `pynacl` as an inline dependency.)

## 2. Settle NAME

`NAME` is the one value with no sensible universal default — it is the site's own identity, both the `mkwp` positional argument and (sanitised by `mkwp` itself) the DDEV project slug. Take it from context when the conversation already names the site unambiguously (an explicit `/kntnt-wp-skills:mkwp <name>` argument, or the operator naming the project in the preceding turns); otherwise ask directly, even under `--yes` — there is nothing to recommend here, so `--yes` cannot skip it silently.

## 3. Resolve the remaining decisions

Present each decision below as a recommendation behind an accept-or-override gate — *"Recommended: X. Accept? [Y/n]"*, the same shape `clone` and `pull` use ([ADR-0005](../../docs/adr/0005-decision-backbone-gates-and-layered-defaults.md)). Walk them in this order (after `NAME`, so `--dirname`'s mirroring recommendation has the settled `NAME` to fall back on) in interactive mode; in `--yes` mode present nothing per gate and accumulate the full record instead. A decision whose recommendation is "omit the flag" needs no operator confirmation of a value — mkwp's own default already is the recommendation.

- **`--dirname`.** Default: `NAME` (mkwp's own default — omit the flag). If the operator names a domain this site is meant to mirror — one that is, or will become, a production site a later `/clone` will reach — recommend `--dirname` set to that domain's full host instead, the same as any other decision here: this dependency check's version floor (`scripts/mkwp_guard.py`, ≥ 1.8.1, [Kntnt/mkwp#3](https://github.com/Kntnt/mkwp/issues/3)) guarantees the local `mkwp` already has the fix for the `--dirname`/`NAME` database-connection defect that, against every `mkwp` ≤ 1.8.0, once made a diverging value fail outright — §4's post-scaffold check below covers whatever else could still make a scaffold fail, as general robustness, not that specific defect. Resolve the host the same way `clone` derives its own directory name: pipe `{ "site": { "home_url": "https://<domain>" } }` to `uv run "${CLAUDE_PLUGIN_ROOT}/scripts/classify.py"` and read `classifications.project_name.directory_name` — the exact function `clone` uses for its own directory naming (issue #11), so a later `/clone` into this same directory needs no rename. Always quote the resolved value in the final command (`--dirname="<value>"`) — the same defense-in-depth `clone` applies against an operator-corrected value carrying a shell metacharacter.
- **`--directory`.** Default: omit (mkwp's own default — the current directory). Recommend an explicit path only when context names one (e.g. the operator mentions a projects folder to create it under).
- **`--title`.** Default: omit (mkwp's own default — `NAME`). Recommend a nicer title only when context supplies one distinct from the technical `NAME` (a company or project name).
- **`--email` / `--user`.** Default: omit both (mkwp's own defaults — current OS username @ hostname, then that email's local part). Recommend the operator's own known email address when the session context supplies one (present the pair together, since `--user` follows from `--email` unless context names a distinct username).
- **`--language`.** Default: omit (mkwp's own default — `en_US`). Recommend a locale only when context indicates one (e.g. a Swedish site or operator → `sv_SE`).
- **`--php`.** Default: omit (mkwp's own default — currently 8.5). Recommend a specific version only when context calls for matching a known target.
- **`--wp`.** Default: omit (mkwp's own default — latest). Recommend a specific version only when context calls for pinning to a known target.
- **`--themes`.** Default: omit (mkwp's own default theme). Recommend only when context names one.
- **`--plugins`.** Default: **Kntnt Extractor** — recommended every run, because the site needs the control channel the moment it becomes a production site `clone`/`pull` reach ([ADR-0016](../../docs/adr/0016-kntnt-extractor-replaces-novamira-as-control-channel.md)); append any plugin context names after it, comma-separated. This is a **recommended default the operator may decline**, not a hard dependency — a site can always add Kntnt Extractor later; if the operator declines the gate, omit it and move on.

  **Resolve the exact URL at run time — never hardcode a version.** Kntnt Extractor has no WordPress.org listing; it is distributed from `https://github.com/Kntnt/kntnt-extractor`. Never a bare repo URL (a web page, not a zip — it fails `wp plugin install` outright) and never a `.git` clone URL. The value that actually works is the **latest GitHub release's packaged zip asset**: resolve it with `curl -sL https://api.github.com/repos/Kntnt/kntnt-extractor/releases/latest`, then select the asset **whose `name` matches `kntnt-extractor-*.zip`** — never `.assets[0]` by position; a checksums or SBOM asset published ahead of the zip in a future release would silently resolve to the wrong file — and read that matched asset's `browser_download_url` (a URL of the shape `https://github.com/Kntnt/kntnt-extractor/releases/download/<tag>/kntnt-extractor-<version>.zip`). Use that resolved URL as the `--plugins` entry — never a literal version pinned in this file, since it goes stale the moment Kntnt Extractor cuts a new release.

  **If resolution fails, never pass a guessed or empty value to `--plugins`.** The unauthenticated GitHub API is rate-limited to 60 requests/hour — a rate-limited response has no `assets` array at all — and the call can fail outright for any ordinary network reason; either failure, or a response whose assets contain no `kntnt-extractor-*.zip` match, means the URL cannot be resolved this run. When that happens, drop Kntnt Extractor from this run's `--plugins` value entirely (never a placeholder or the last-known URL), tell the operator resolution failed and why, and recommend they either retry the gate shortly or add Kntnt Extractor manually afterwards once they have a working URL (`ddev wp plugin install <url> --activate`) — a `--plugins` entry that is not a real zip URL fails `wp plugin install` and can abort the whole `mkwp` run over something this skill could have avoided. A graceful drop is always preferable to breaking the scaffold over a recommended-but-optional plugin.
- **`--mu-plugins`.** Default: omit (mkwp's own default — none). Recommend only when context names one.

**Never gather a password.** No `--password` flag is ever offered or passed at any gate — `mkwp`'s own random 16-character generation is always what creates the first user. **The generated value must never enter this skill's context either** — see §4, which redirects the whole run to a log file for exactly this reason; only the log file (or the operator's own terminal, if they run `mkwp` themselves) ever shows it.

## 4. Build and run

Assemble the resolved flags into one `mkwp` invocation — `mkwp <name> [--dirname="<value>"] [--directory="<value>"] [--title="<value>"] [--email="<value>"] [--user="<value>"] [--language="<value>"] [--php="<value>"] [--wp="<value>"] [--themes="<value>"] --plugins="<value>" [--mu-plugins="<value>"]` — omitting every flag whose resolved value is mkwp's own default, so the command never carries a redundant flag. Quote every value.

**Redirect the run to a log file — never let its stdout/stderr flow through this skill's captured tool output.** Every successful `mkwp` run ends by printing the generated admin username and password verbatim (`Username: $user\npassword: $password`) to its own stdout; if that output is captured as the Bash tool's result the way an ordinary command's would be, the password rides straight into this skill's context, which is exactly the leak the password clause above forbids. Pick a fresh log path the operator can find afterwards (e.g. `mktemp -t mkwp-run` in the operator's current directory) and run `mkwp <name> [flags...] > "<log-file>" 2>&1` — from the operator's current directory (or the resolved `--directory`) — so the full report lands on disk instead of in this conversation.

`mkwp` performs its own `ddev config`, first `ddev start`, WordPress install, plugin/theme installation, and `ddev launch` — there is nothing downstream in this skill that needs to correct or restart the engine the way `clone`'s bookend does, because there is no production database version to pin against.

Confirm the scaffold actually succeeded before reporting success with a plain file-existence check — never a content read: does `wp-config.php` now exist in the resolved directory? If it does, stop there; do not open `<log-file>` to "double-check" — the credentials block is in it. If it does not exist, the scaffold failed — general robustness against whatever could still make `mkwp`'s own run die partway (the version floor in §1 already rules out the `--dirname`/`NAME` defect specifically), not a diagnosis of one named defect. Clean up before reporting: from the resolved directory (if it exists), run `ddev delete -Oy` to unregister the DDEV project and drop its throwaway database, then remove the directory if it holds nothing but `mkwp`'s own placeholder scaffold (never real content), so a retry starts clean. Never retry automatically — the operator decides whether and how to retry. Do not `cat` `<log-file>` whole to explain the failure — a run that reached far enough to print credentials before a later step failed would still have written them; `grep` for a failure marker instead if a specific diagnosis is warranted.

## 5. Report

Report the full decided-and-done record: every resolved flag and where it came from (context-derived, gate-accepted, or left at mkwp's own default), the site's local URL (`<name>.ddev.site`), the directory it landed in, and the path to `<log-file>`. Point the operator at that log file (or their own terminal, if they ran `mkwp` themselves) for the generated admin username and password — `mkwp`'s own full report is waiting there, unread by this skill, which is why this skill can truthfully say it never captured or repeats it. Once the operator has copied the credentials elsewhere, they may delete `<log-file>` themselves; it is not otherwise cleaned up here, since a failed run's log is the operator's evidence for the `mkwp` defect diagnosed above. If Kntnt Extractor was installed, remind the operator it still needs to be **activated**, and that a later `/clone` or `/pull` will need an **Application Password** for a WordPress user holding both the `kntnt_extractor_operate` and `manage_options` capabilities before it can reach the site ([ADR-0016](../../docs/adr/0016-kntnt-extractor-replaces-novamira-as-control-channel.md)).

## Testing note

The version guard's pass/fail logic is unit-tested at its own seam (`scripts/mkwp_guard.py`), and the docs-consistency test binds this manual page's flags to the plugin's flag registry. The orchestration prose itself — driving `mkwp` and reading its own report back — is a **human-verified residual** ([spec](../../docs/spec.md) *Testing Decisions*), the same posture `clone` and `pull` take for their own real-tool interactions. Nothing in this file reaches a live `mkwp` invocation during the automated suite.
