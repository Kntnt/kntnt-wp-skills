# pull

Refresh an existing local DDEV copy of a production WordPress site. `pull` and its sibling `clone` run **one shared transfer engine** — discovery, extraction on production, download, verification, remote cleanup, import, and localisation — and differ only at the bookends. A clone is a pull with no baseline; a pull always has one, so only the delta moves: it takes a rollback backup of the local database first, diffs production against the stored baseline so only new and changed files are extracted, and re-applies the local state — the inactive plugin set, the object-cache drop-in — afterwards. Every run begins with a **health check** that fails early and cheaply, then walks the operator through a series of **gates** whose recommendations come from the deterministic helpers; `--yes` runs the whole thing unattended and prints a full record. A **risk warning** is always emitted before the destructive local steps.

Read `docs/spec.md` (the specification), `CONTEXT.md` (the glossary — its terms are used verbatim below), and `docs/implementation-notes.md` (the invocation-level literals) alongside this file. Where a literal here and the spec diverge, the spec wins.

## 0. Help gate

If the arguments are `help`, `--help`, or `-h`, run `uv run "${CLAUDE_PLUGIN_ROOT}/scripts/help.py" pull`, emit its output verbatim as Markdown, and stop. Do nothing else — no control-channel call, no file operation.

## How the engine works

**Control channel.** Production is reached **solely** through the [Kntnt Extractor](https://github.com/Kntnt/kntnt-extractor) plugin's REST API on the live site — never SSH ([ADR-0016](../../docs/adr/0016-kntnt-extractor-replaces-novamira-as-control-channel.md)). It is reached over HTTPS, authenticated as a real WordPress user via an Application Password (HTTP basic auth); that user's own capabilities bound what any call can reach. Authorisation is a **flat two-capability gate** — `kntnt_extractor_operate` is the on-switch and `manage_options` authorises the data; a user with one but not the other reaches nothing. The surface is fixed and minimal: `GET /status` (the unauthenticated API-version handshake), `GET /environment`, `GET /tables`, `GET /files`, `POST /extractions`, `GET /extractions`, `GET /extractions/{id}`, `POST /extractions/{id}/consume`, and `DELETE /extractions/{id}`, plus the `manage_options`-only `GET /audit-log`. There is no arbitrary PHP or WP-CLI execution on production and no SSH — the plugin owns the outside-docroot extraction, the per-segment sealing, the one-time download link, and the TTL cleanup, so the engine generates no shell script, handles no passphrase, and stages nothing in the docroot ([ADR-0017](../../docs/adr/0017-discovery-over-extractor-rest-two-phase.md)). Never deactivate or delete Kntnt Extractor on production; it is the control channel. The skills **pin Extractor to API version ≥ 2** via `GET /status` and refuse an older install with a precise upgrade instruction.

**The deterministic helper seam.** Every computation that needs neither production nor DDEV is done by a helper script under `${CLAUDE_PLUGIN_ROOT}/scripts/`, invoked with `uv run` and fed JSON on stdin. **Never compute a diff, a classification, a resolved plan, an extraction selection, an unseal, or a dump verdict by hand** — the model orchestrates gates and REST calls, the helpers decide the shapes ([ADR-0005](../../docs/adr/0005-decision-backbone-gates-and-layered-defaults.md)). The seam is:

- `scripts/discovery.py` — assembles the one canonical discovery document from the four Extractor sources (`{ environment, tables, files, bootstrap }`).
- `scripts/bootstrap_parse.py` — parses the reassembled bootstrap extraction's `.sql` into the row-level signals (attachment metadata, entity counts, mass-send scan) the `environment`/`tables`/`files` surface cannot supply.
- `scripts/classify.py` — turns that document into the table split, the define classes, the flagged blobs, the thumbnail exclude-set, and the derived project and directory names.
- `scripts/resolve_plan.py` — resolves the ordered decision list over the layered defaults (`resolve`), and reduces an accepted plan back to the saved plan (`save`).
- `scripts/build_exclusions.py` — the single deterministic assembler of the **resolved exclusion set**: unions its canonical `ALWAYS_EXCLUDED` constant (the configuration file and its credential-bearing backup/swap/variant siblings, `.env` files anywhere in the tree, root-level SQL dumps and key material, drop-ins, logs, caches, upgrade dirs) with the decision-gated thumbnail, blob, and media exclusions, so the baseline diff and the selection are fed one byte-identical set — never a hand-assembled list (issue #35).
- `scripts/filter_manifest.py` — restricts production's unfiltered `GET /files` walk to the resolved exclusion scope, locally, so the exclusion set never travels to production (issue #18).
- `scripts/baseline_diff.py` — diffs production's current (locally-filtered) manifest against the stored baseline and returns the new/changed set to extract and the production-deleted set the deletion gate draws from.
- `scripts/build_selection.py` — turns the resolved table split and file set into the `{ tables, tables_structure_only, files }` a `POST /extractions` submits, refusing an overlapping or empty selection locally (platform constraint 1).
- `scripts/unseal.py` — generates the run's ephemeral X25519 key pair (`keygen`) and opens a downloaded `KNTNTEXT` sealed container into the importable `.sql` and its files (`unseal`).
- `scripts/dump_sanity.py` — verdicts the reassembled dump against the discovered prefix before the import.

The only production-side template that remains is the local capture mu-plugin `templates/kntnt-wp-skills-mailpit.php`, dropped into the **local** copy when the mail decision resolves to capture — it never travels over the control channel.

**Run modes.** Interactive walks each gate. `--yes` accepts every recommendation, never pauses, and prints the full decided-and-done record. **Replay** engages automatically when a saved plan exists: interactive collapses to a single *Replay the saved plan?* gate, `--yes` replays silently — except the mass-send valve, which re-surfaces the mail gate even on a silent replay when a freshly-poised campaign would otherwise be masked ([ADR-0009](../../docs/adr/0009-live-mail-default-with-mass-send-valve.md)).

**The gate shape.** Every decision is a **recommendation** behind an accept-or-override gate — *"Recommended: X. Accept? [Y/n]"*. `Y` accepts; `n` reveals the alternatives. Even multi-valued decisions take this shape. The recommendation is whatever `scripts/resolve_plan.py` resolved for that decision; a declined gate records the operator's answer, which the resolver layers above the recommendation.

**Persistent config.** Two per-project files at the local project root: the saved plan `.kntnt-wp-skills.json` (settled per-site answers, committed, all keys optional) and the derived, gitignored `.kntnt-wp-skills/` (baseline manifest `last-sync.json`, rollback backups, trash). The saved plan stores **decisions, never computed lists**, so nothing in it goes stale.

**Subagent delegation.** Four phases — `discovery-classify` (Sonnet, low reasoning effort), `extract-transfer` (Sonnet, medium), `manifest-baseline-diff` (Haiku, low), and `thumbnail-smoke-test` (Haiku, low) — run in pinned subagents defined under `${CLAUDE_PLUGIN_ROOT}/agents/`, so the REST round-trip logs, curl/download output, and thumbnail-regeneration warning spam they produce never enter this agent's own context. Everything else — the health check, every gate, the risk warning, plan resolve, the wp-config edits, and the final report — stays here: subagents run once against a compact JSON task envelope and can never ask the operator anything. Each subagent writes any large payload (raw discovery JSON, the manifest, the reassembled dump) to a scratchpad file and returns only a short summary, the file paths, and its **evidence block** — exit codes, artifact paths with their SHA256, row/file counts, and a `DONE`/`FAILED` marker. **A result missing its evidence block is treated as `FAILED`, whatever its prose claims.** Verification stays deterministic, never a second LLM call: before trusting a subagent's result, re-run the one or two cheap checks its evidence block names — the dump-sanity verdict on the reassembled `.sql`, a `wp db check` exit code, or similar.

## 1. Health check

Mandatory step 0 of the engine, before any heavy work. On any failure, abort with a precise remediation message — never a stack trace.

1. **Verify dependencies.** Before any production call, confirm every local and production dependency this run needs — local first, since it is cheaper to fail before ever reaching production. On the first missing dependency, stop immediately: report what to install or fix, where to get it, and the exact command to re-run once done. The agent never installs system software itself; where a fix is safely agent-runnable (e.g. `brew install jq`), offer to run it as its own accept-or-override gate before re-checking, rather than assuming consent. This gate is never auto-accepted under `--yes` — an unattended run has no operator present to consent to installing system software, so `--yes` aborts with the remediation message instead of running the fix.
   - **Local.** `ddev` is on `PATH` and its container backend actually responds — `ddev version` exits zero, and a cheap Docker/Colima liveness probe (`docker info`, else `colima status`) also succeeds; a `ddev` binary with a stopped backend otherwise fails silently deep inside the first `ddev start`. Every required CLI tool is on `PATH`: `uv`, `jq`, and `curl` (the sealed-container unseal is a `uv`-run helper with `pynacl` as an inline dependency, so no separate encryption or checksum binary is required).
   - **Production.** `GET /status` on the configured Kntnt Extractor endpoint for the target production URL, proving it is reachable and reports **Extractor API version ≥ 2** — the version that ships the `environment` endpoint, structure-only extraction, and caller job listing together. An absent, unreachable, or too-old Extractor fails here with a precise install-or-upgrade instruction. If several endpoints could be the target, or the configured base URL does not match the intended production URL, ask which to use rather than guess.
2. **Prove the channel is live, correctly targeted, and authorised.** `GET /environment` on the configured Application Password, in one round trip: a `200` is the liveness proof; its `home_url` proves the channel **targets production**, not the local DDEV site (the verify-targets-prod safety rail — it must match the intended production URL, and if it does not, ask rather than guess); and a `200` proves the user holds **both** `kntnt_extractor_operate` and `manage_options`, since `/environment` is gated on both. A `403` fails fast with a precise, per-capability remediation, disambiguated where useful by whether `GET /audit-log` — which is `manage_options`-only — also refuses (a `403` from `/environment` but a `200` from `/audit-log` means the `kntnt_extractor_operate` capability is missing).
3. **Sweep stranded jobs.** `GET /extractions` lists the caller's own non-terminal jobs, and `DELETE /extractions/{id}` cancels each — leftovers of an aborted earlier run, belt-and-braces with the plugin's own TTL cleanup. Runs before the preflight (step 4) so a batched pair of calls can never cancel the preflight's own probe job; this sweep must never run concurrently with an in-flight preflight.
4. **Preflight the download path.** Submit a throwaway minimal extraction — `POST /extractions` for **exactly two** tiny structure-only tables and no files, `{table_prefix}options` and `{table_prefix}users` with `table_prefix` taken from step 2's `GET /environment` response (both tables exist in every WordPress install, and a structure-only dump is DDL-only, so table size is irrelevant), sealed to a fresh `uv run scripts/unseal.py keygen` public key — poll it to ready via `GET /extractions/{id}` under the standard poll discipline (15 s cadence, 120 s per-request timeout, retry on transport failure; the preflight's overall budget is 10 minutes), fetch its one-time `download_url` over HTTPS from the local side, unseal it with `scripts/unseal.py`, then **consume** the job with `POST /extractions/{id}/consume`. This exercises the real serving path — permissions, extension rules, basic auth, WAF/CDN behaviour — before the heavy extraction; the one-time link commonly 404s or is WAF/CDN-blocked on managed hosts. Two tables mean exactly two packaging chunks, so the probe also exercises the continuation path — a one-chunk job completes on the create-time kick alone and never touches it, which is exactly how a dead continuation loopback stays hidden until mid-run ([ADR-0018](../../docs/adr/0018-poll-discipline-and-two-chunk-preflight.md)). Verdict by measured wall time from the `201` to the first `ready` poll: **≤ 90 s** passes silently; **slower but ready within the 10-minute budget** passes with a loud warning and an operator gate — *"This host advances extraction jobs at backstop cadence (~N s per chunk, measured by the preflight); the heavy extraction may be impractically slow. Upgrade Kntnt Extractor to ≥ 0.2.1 (Kntnt/kntnt-extractor#18/#19); if it is already ≥ 0.2.1, the host's PHP worker can neither detach from the client nor reach its own loopback URL — fix the host's loopback, or expect the job to advance only one 15 s packaging budget per cron cycle."* — then ask the operator whether to continue or abort; continuing can be legitimate (a small site, or a ≥ 0.2.1 host whose watchdog packages a full budget per cycle), but the operator must own that choice, so under `--yes`, with no operator present to own it, the slow verdict aborts with the warning as the remediation. **Not ready within the 10-minute budget** — the poll discipline's stall rule applies inside it too — aborts the health check with that same message as the precise remediation.

## 2. Discovery

Discovery is reconstructed client-side from three sources, in order, and split across two extraction phases because the classification that shapes the heavy extraction must be computed *before* that extraction exists ([ADR-0017](../../docs/adr/0017-discovery-over-extractor-rest-two-phase.md)). The **database password and the other production secrets never enter model context** — the `environment` endpoint masks the secret define family server-side, and the database is never fetched in cleartext.

1. **`GET /environment`** supplies the runtime and configuration scalars: home and site URLs, content and uploads paths, the WordPress core version, the table prefix, the PHP version (to pin DDEV's at major.minor), the database server flavour/version/collation (MySQL 8 vs MariaDB, to pin DDEV and avoid the collation import crash), the active plugins and whether a multilingual plugin is among them, the drop-ins present, and the resolved `wp-config` defines with the secret family already redacted.
2. **`GET /tables` and `GET /files`** supply the sizes and enumerations: the authoritative table list with per-table row-counts and byte sizes, and the whole content tree (path + size + mtime, install-root-relative). `GET /files` is **paged via an opaque `cursor`** — loop, following the cursor until it is null, and flatten the pages into one manifest.
3. A cheap **bootstrap extraction** — a normal `POST /extractions` of `wp_posts`, `wp_postmeta`, `wp_users`, the active recognised-mailer tables, and Action Scheduler, and no files, sealed to a fresh `scripts/unseal.py keygen` public key — is polled to ready via `GET /extractions/{id}`, downloaded, reassembled with `scripts/unseal.py unseal`, parsed with `uv run scripts/bootstrap_parse.py` (input `{ sql_path, table_prefix }`), then **consumed** with `POST /extractions/{id}/consume`. It yields the three signals that need row-level data: the generated-thumbnail exclude-set (from `_wp_attachment_metadata`), the verify-phase entity counts (`wp_users` is carried here so `entity_counts.users` has a source), and the mass-send poised-campaign scan. The bootstrap is small and is consumed before the main extraction starts, so Extractor's one-active-job rule is never contended.

Assemble the three sources plus the bootstrap parse into one envelope — `{ "environment": <GET /environment>, "tables": <GET /tables>, "files": <flattened GET /files manifest>, "bootstrap": <bootstrap_parse.py output> }` — and pipe it to `uv run scripts/discovery.py`; it writes the **canonical discovery document** (redacting any secret define value again at the boundary as a second line of defence). Pipe that document to `uv run scripts/classify.py` to get the classifications: the full-data / empty table split (with form-submission tables tagged their own `user_submissions` category — [ADR-0014](../../docs/adr/0014-user-submissions-own-gate-default-empty.md)), the portable vs auto-excluded defines, the flagged heavy blobs, the thumbnail exclude-set, and the derived project and directory names. If either helper exits non-zero, stop and report its stderr diagnostic — a malformed scan never rides into the run.

**Delegate this phase to `discovery-classify`.** The subagent makes the `GET /environment`, `GET /tables`, and paged `GET /files` calls, runs the bootstrap extraction (submit, poll, unseal, `bootstrap_parse.py`, consume), pipes the assembled envelope through `scripts/discovery.py` and `scripts/classify.py`, writes the canonical discovery document and the classifications to scratchpad files, and returns only their paths, a one-line summary (table, define, and blob counts; the derived project name), and its evidence block: each helper's exit code, the SHA256 and byte size of both written files, the discovered table and active-plugin counts, and a `DONE`/`FAILED` marker. Re-read the written discovery document yourself before resolving the plan against it — a `FAILED` marker or a missing evidence block stops the run here, before any gate is presented.

## 3. Resolve the plan and walk the gates

Assemble the resolve envelope — `{ "operation": "resolve", "skill": "pull", "flags": [...], "discovery": <document>, "classifications": <classifications>, "saved_plan": <.kntnt-wp-skills.json or null>, "answers": {} }` — and run `uv run scripts/resolve_plan.py`. It returns the run mode, whether this is a replay, the ordered **gate** list, and for every decision its **recommendation**, resolved value, and source layer (built-in < live derivation < saved config < this-run answer, with a flag pinning above all four; `--yes` stops at the saved-config layer). The pull decision list carries the bookend decisions clone does not: the preserved inactive plugin set, the object-cache mode, and deletion mirroring. Both skills' decision lists also carry a `user_submissions` gate for form-entry tables the classifier tagged privacy-sensitive — carry/empty, default **empty** ([ADR-0014](../../docs/adr/0014-user-submissions-own-gate-default-empty.md)). Its resolved choice is folded into `db_table_content`'s own resolved value and recommendation — a carry moves the tagged tables from that decision's empty list into its full-data one — so `db_table_content` is always the one list `scripts/build_selection.py` builds the extraction's `tables` / `tables_structure_only` split from; this is exactly why the walk re-runs `resolve_plan.py` after every answered gate, so a `user_submissions` answer given after `db_table_content` was already walked still folds into the selection that is extracted.

Walk the returned gates in order. For each, present the recommendation and accept on `Y`; on `n`, reveal the alternatives and record the operator's answer, then re-run `scripts/resolve_plan.py` with that answer in `answers` so the resolved value and every dependent recommendation stay consistent. In `--yes` mode present nothing — every recommendation is accepted — and accumulate the record. When a saved plan exists, present only the single *Replay the saved plan?* gate (plus the mail gate if the valve re-surfaced it).

The mail decision leads with its `findings`: when the mass-send valve flipped, the gate opens with the loud, specific warning naming the engine, the campaign, and the recipient count. Once the walk is done, persist the accepted plan by running `scripts/resolve_plan.py` with `{ "operation": "save", "resolved": <resolved plan>, "saved_plan": <.kntnt-wp-skills.json or null>, "source": { "extractor_endpoint": <the health-checked Extractor endpoint's base URL>, "live_url": <discovery `home_url`> } }` and writing the result to `.kntnt-wp-skills.json` — the settled decisions and the persisted `source` (the Extractor endpoint and live URL, stored opaquely by `resolve_plan.py`), never the computed lists. Pass the prior saved plan back so the save carries forward any key a pull does not walk — notably the DDEV `target` a preceding clone recorded, and the `source` if this run does not resupply it — rather than dropping it on the re-save.

## 4. Pull bookends — verify the prefix, diff the baseline

The pull-only setup over the local site that already exists. Both steps run before the extraction: the prefix guard fails cheaply, and the diff decides what the extraction even contains.

- **Verify the table prefix.** Read the local table prefix from the marked block in `wp-config.php` and compare it to production's `table_prefix` from discovery. On a **mismatch, abort before any extraction or import** — importing production's tables under a prefix the local install does not read leaves WordPress finding zero tables (platform constraint 12). This is the pull manual page's before-import guarantee; because the local prefix already exists and persists, the later marked-block step writes no prefix at pull — it only relies on the match this step already proved.
- **Assemble the resolved exclusion set.** Pipe `{ "classifications": <the classifications `scripts/classify.py` produced>, "plan": <the resolved plan `scripts/resolve_plan.py` returned> }` to `uv run scripts/build_exclusions.py` — **never assemble the set by hand**. It unions its canonical `ALWAYS_EXCLUDED` constant (the configuration file and its credential-bearing backup/swap/variant siblings, `.env` files anywhere in the tree, root-level SQL dumps and key material, drop-ins, logs, caches, upgrade dirs — the single source of truth for the always-excluded paths) with the decision-gated thumbnail, blob, and media exclusions and emits `{ "exclusions": [...] }`. This one set is what the diff below filters the manifest against and what the next clone/pull baseline is taken under, so a scope change can never poison the deletion diff (issue #35).
- **Diff the baseline.** Read the stored baseline `.kntnt-wp-skills/last-sync.json` (path → size + mtime, and the scope it was taken under). Fetch production's **whole** content tree with **`GET /files`** — paged via the opaque `cursor`, looping until it is null — unfiltered, so the exclusion set never travels to production, a small request whatever its size (issue #18). Filter it locally: pipe `{ "entries": <the flattened `GET /files` manifest's entries>, "unreadable": <any subtree the walk could not read, else an empty list>, "exclusions": <the resolved exclusion set — the `scripts/build_exclusions.py` output above, never a separately-assembled list> }` to `uv run scripts/filter_manifest.py`, which restricts it to the in-scope entries and attaches the resolved set as its scope; it aborts loudly instead if the manifest reports any unreadable directory, since a silently-incomplete tree cannot be trusted for the deletion gate. Combine the stored baseline with that locally-filtered manifest into `{ "baseline": <stored last-sync.json>, "current": <locally-filtered manifest> }` and run `uv run scripts/baseline_diff.py`; it returns `new_or_changed` (the set to extract and pull) and `production_deleted` (the set the deletion gate draws from), the deletion set already restricted to paths still in scope this run (the scope-intersection rule, [ADR-0006](../../docs/adr/0006-baseline-manifest-diff-with-scope.md)). **Never diff the local filesystem** — local mtimes are unreliable through the archive-and-sync chain, so both sides of the diff are production mtimes (platform constraint 19). Keep the locally-filtered current manifest: it becomes this run's new baseline once localisation succeeds.

Only `new_or_changed` is extracted and transferred, but **the database is always dumped in full** — trimmed by the empty-table classification it is small and not worth diffing.

**Delegate this phase to `manifest-baseline-diff`.** Fetching production's unfiltered `GET /files` tree, scope-filtering it locally via `scripts/filter_manifest.py`, and running `scripts/baseline_diff.py` against the stored baseline all run in the `manifest-baseline-diff` subagent: it writes the locally-filtered current manifest and the diff helper's stdout (the `new_or_changed` and `production_deleted` sets) to scratchpad files, and returns their paths, the `new_or_changed` and `production_deleted` counts, and its evidence block: the manifest's row count before and after scope filtering, the helper's exit code, the SHA256 of both written files, and a `DONE`/`FAILED` marker. Read the returned diff-output path for the actual `new_or_changed` and `production_deleted` sets — the extraction step below carries the former, the deletion gate itemises the latter. Confirm the manifest's row count structurally against the evidence block's `manifest_scoped_rows` — e.g. `jq '.entries | length' <manifest path>` — before trusting the diff; the manifest is single-line JSON, so `wc -l` on it proves nothing.

## 5. Extraction on production (background job)

Build the extraction's **selection** from the resolved plan — **never assemble it by hand**. Pipe `{ "table_content": <the resolved plan's `db_table_content` decision, its `full` and `empty` lists — never the classifications' raw table split, which `resolve_plan.py` has already folded with the resolved `user_submissions` gate (ADR-0014)>, "files": <the `new_or_changed` paths read from the diff-output file the `manifest-baseline-diff` subagent wrote in step 4, not the whole tree> }` to `uv run scripts/build_selection.py`; it emits `{ tables, tables_structure_only, files }` — content/CRM tables carried in full, every empty-classified table as structure-only DDL, the new/changed file set — and refuses a self-overlapping or empty selection locally (platform constraint 1). The file set is an explicit list of paths already in scope: `new_or_changed` was diffed against the manifest under the resolved exclusion scope (step 4), so no command-line exclusion patterns or argument-limit hazards arise, and the database is still submitted **in full** (`tables` / `tables_structure_only`), trimmed only by the empty-table classification.

Generate the run's ephemeral key pair with `uv run scripts/unseal.py keygen` — it writes the raw private key to a file that never leaves this machine and prints only the base64 `public_key`. Submit the selection with **`POST /extractions`**, carrying that `public_key`; the plugin returns `201 { id, state: "queued" }` and runs the extraction as its own detached background job **outside the docroot**, sealing each table and file segment into the `KNTNTEXT` container. No passphrase is ever generated, fetched, or web-served — the data is sealed to the run's **ephemeral** public key, and only the never-transmitted private key can open it; the database dump, the live-safe read, the sealing, and the outside-docroot staging are all the plugin's responsibility, not the skill's.

**Poll** the job by id — `GET /extractions/{id}` — every 15 s after a successful poll, with a 120 s per-request timeout, until `state == "ready"` and `download_url` is non-null; the orchestrator passes `poll_max_wait_seconds: 3600` — the overall wall-clock budget — to `extract-transfer`. On a transport timeout, connection error, or 5xx: log it and retry after 30 s (60 s from the second consecutive failure), resetting to the 15 s cadence on the next success — a single bad response is never failure. The `progress` counters are reported between polls (e.g. `tables 5/12, files 40/312`), so a slow-but-advancing job is visibly distinct from a wedged one; the job counts as advancing when its `state` changed or the sum `progress.tables_done + progress.files_done` increased, and structure-only tables count toward that progress. Failure is exactly: `state == "failed"` (surfacing the `error` the plugin reports), a vanished job (`404`), no advance within the 10-minute stall window, or exhaustion of the overall budget — aborting rather than hanging, and never on transport errors alone ([ADR-0018](../../docs/adr/0018-poll-discipline-and-two-chunk-preflight.md)).

## 6. Download, verify, and unseal

Fetch the one-time `download_url` over HTTPS with `curl -fSL -C - --retry 3` (resume and retry) into the local scratch area, then open it with `uv run scripts/unseal.py unseal` and the run's private key: each segment's sealed key is opened (`crypto_box_seal`), each segment decrypted (`crypto_secretbox`), and the container reassembled — the table segments concatenated into one importable `.sql` with a prepended connection-safe preamble (charset `SET NAMES`, `FOREIGN_KEY_CHECKS`, SQL mode), the file segments written to disk by their install-root-relative path. The `crypto_secretbox` **authentication is what catches a truncated or corrupted download** — a tampered or short segment fails to open, so a bad transfer is caught before it touches the local site; **no separate checksum step is needed**.

## 7. Close the exposure window

Immediately after the container unseals, **consume** the job with `POST /extractions/{id}/consume` and confirm it is gone — the happy-path close that deletes the artifact on production. The plugin's own TTL/watchdog cleanup and the next health check's stranded-job sweep are the backstops; this explicit consume keeps the exposure window to minutes even so. `DELETE /extractions/{id}` is **only** for cancelling a stranded or aborted job (the health-check sweep), never the happy-path close.

**Delegate this phase to `extract-transfer`.** Steps 5–7 — building the selection, `keygen`, submitting the extraction, polling to ready, downloading, unsealing, and consuming the job — run in the `extract-transfer` subagent: it writes the reassembled dump and the unsealed files to scratchpad paths and returns only those paths, the polled job state, and its evidence block: the job id and terminal `state`, the reassembled-dump and unsealed-files byte sizes, the SHA256 of the reassembled `.sql`, and confirmation that the job was consumed. Re-run `scripts/dump_sanity.py`'s verdict on the reassembled `.sql` yourself before treating the extraction as trustworthy — an evidence block claiming `DONE` against a failed dump-sanity verdict is treated as `FAILED` regardless.

## 8. Risk warning (always emitted)

Before the destructive local steps, always emit the risk warning itemising the copy's outward-reaching behaviours:

- The resolved mail mode — a live mailer can send real mail.
- A running cron — can fire real webhooks, capture real payments, post to connected social accounts, or re-validate a licence from the dev domain.
- The real user data now in the database.
- **Mandatory:** each per-submission form-to-service integration `classifications.integrations.form_to_service` detected — a form plugin's active service add-on (e.g. WS Form's Mailchimp add-on) fires on a single submit and is invisible to the mass-send valve, which only watches a poised bulk campaign ([ADR-0009](../../docs/adr/0009-live-mail-default-with-mass-send-valve.md)). List every finding's `warning` verbatim, one per bullet — e.g. "submitting form WS Form locally writes to live service Mailchimp" — so submitting a form on the local copy is never mistaken for harmless.

Interactive waits for the operator's confirmation to proceed; `--yes` prints it for the record and proceeds.

## 9. Import and localise

Local and destructive. Run in this exact order — the order is a safety rail ([spec](../../docs/spec.md) *Import and localise*):

1. **Rollback backup (pull only, before anything destructive).** Back up the local database with `ddev export-db` to the durable, gitignored `.kntnt-wp-skills/backups/local-pre-import-<timestamp>.sql.gz` **before any destructive local step** — a bad refresh is then one import away from undone. Keep its path for the report (review point 18 in `docs/implementation-notes.md`).
2. **Sanity-check the dump.** Run `uv run scripts/dump_sanity.py` with the discovered prefix and the resolved plan's `db_table_content` empty-table list (the same folded list step 5's selection carried, not the classifications' raw split) against the reassembled dump: the content table created under the prefix, its inserts present, every one of those tables created but empty. Abort on any failure — a wrong-prefix dump leaves WordPress finding nothing.
3. **Import.** `ddev import-db` the reassembled dump; verify table and post counts afterwards.
4. **Merge the unsealed files.** `scripts/unseal.py` wrote the extraction's file segments to a staging tree by their install-root-relative path; merge that tree over the content directory as a **merge**: the local `wp-config.php` and local-only files survive; production's plugin, theme, and media files in the new/changed set overwrite.
5. **Deletions, if enabled (off by default).** Deletion mirroring is **off by default** ([ADR-0010](../../docs/adr/0010-deletion-mirroring-opt-in-trash.md)); when the remembered per-site answer is Yes, itemise the two sources — the `production_deleted` set read from the diff-output file the `manifest-baseline-diff` subagent wrote (step 4) and the plugin/theme **drift** (local plugins/themes with no production counterpart, as a checklist) — and move confirmed items to the timestamped local trash `.kntnt-wp-skills/trash/<timestamp>/`, **never a hard `rm`**, reversible until emptied, and report the path. Under `--yes` nothing is ever removed **without a saved Yes** already on record — a `--yes` run never turns a No, or an unanswered default, into a deletion.
6. **Object-cache drop-in (pull only).** Resolve the drop-in by the **ownership rule** — no local drop-in → nothing; a different owner than production → keep local; the same owner → take production's — then **verify a real request succeeds** and **auto-remove** the object-cache drop-in on failure, reporting it. A production drop-in pointing at a loopback cache host is fatal against DDEV (platform constraint 16).
7. **Write the marked block.** Write the chosen portable defines into the marked block the skills own in `wp-config.php` (the prefix was already verified in step 4 and is not rewritten at pull). **Surface** in the report any **new production define** of the portable class the classifier found — configuration drift brought to attention rather than silently ignored.
8. **Apply mail and cron.** For the resolved mail mode, either keep the existing mailer or install the capture mu-plugin `templates/kntnt-wp-skills-mailpit.php` into `wp-content/mu-plugins` (it short-circuits `wp_mail` at top priority to DDEV's Mailpit, catching API mailers that never touch sendmail). For cron, `--no-cron` writes `define('DISABLE_WP_CRON', true);` into the marked block; otherwise leave it running.
9. **Re-apply the preserved inactive set (pull only).** Re-apply the **preserved inactive** plugin set — derived from live local state this run — with `ddev wp plugin deactivate <list> --skip-plugins --skip-themes`, so plugin and theme code is **skipped during deactivation** and an object-cache plugin cannot re-drop its drop-in mid-step. Tools you keep off stay off.
10. **URL-scoped search-replace.** One pass per source URL form, each `ddev wp search-replace '<old>' '<new>' --all-tables --skip-columns=guid --report-changed-only --skip-plugins --skip-themes`. The forms, in order: `https://www.<domain>`, `http://www.<domain>`, `https://<domain>`, `http://<domain>`, the protocol-relative `//www.<domain>` and `//<domain>`, the **escaped-slash** forms `https:\/\/www.<domain>`, `https:\/\/<domain>`, `http:\/\/www.<domain>`, `http:\/\/<domain>` that page builders store inside JSON, the **escaped-slash protocol-relative** forms `\/\/www.<domain>` and `\/\/<domain>` (a stored protocol-relative URL has no scheme to anchor the scheme-ful escaped passes), the **double-escaped-slash** forms `https:\\/\\/www.<domain>`, `https:\\/\\/<domain>`, `http:\\/\\/www.<domain>`, `http:\\/\\/<domain>` that JSON-within-JSON storage (a plugin's serialised config nested inside another plugin's JSON option) produces — `\\/\\/` is not a substring of `\/\/`, so the single-escaped pass alone leaves it untouched — and, by the same argument, the **double-escaped-slash protocol-relative** forms `\\/\\/www.<domain>` and `\\/\\/<domain>`. **Never the bare domain** — it corrupts `user@domain` email addresses. Serialised objects wp-cli safely skips keep the old domain; harmless.
11. **Set the URLs.** Set the `home` and `siteurl` options explicitly to the DDEV URL.
12. **Regenerate thumbnails (metadata-driven delta, regenerable names only).** `ddev wp media regenerate` scoped to the affected attachment IDs — the **metadata-driven delta** at pull: compare each attachment's registered sizes against the files on disk, but an attachment only joins the regeneration set when a **regenerable-named** size is missing — a name `wp media regenerate` would itself derive from the attachment's *current* attached file, `<stem>-<W>x<H><ext>` (the same rule `scripts/classify.py::thumbnail_exclude_set` applies to the transfer exclusion). **Big-image attachments (#30):** when the attached file's stem ends in the exact, case-sensitive terminal token `-scaled` — WordPress's big-image-size-threshold feature scales an over-sized upload down to `<stem>-scaled<ext>`, but generates every registered sub-size from the *pre-scaled* original — a size regenerable from the stem with that token stripped also counts as regenerable-named, additive to the unstripped-stem match, so these attachments' genuine gaps join the regeneration set exactly as any other attachment's would. An attachment whose only gaps are **non-regenerable** names (production filename drift from a historical WebP conversion, a dedup-suffixed PDF preview, or any other name regeneration can't reproduce) must **never** be fed to `wp media regenerate` — default regenerate deletes-then-rebuilds, so it would delete the transferred drifted file and rebuild it under the wrong (new) name, reintroducing the very 404s the wider transfer scope (#26) was fixed to avoid. This is why change #26 also narrowed the transfer exclusion: after it, a regenerable-named gap should be rare (the file was excluded and needs rebuilding), while a non-regenerable-named gap means the file simply wasn't excluded and already travelled — nothing to regenerate. `--regenerate-all` remains the escape hatch that forces the lot, unfiltered. **Delegate this phase to `thumbnail-smoke-test`**, which runs the delta regeneration, swallows its per-image progress and warning spam, and returns only the regenerated-attachment count, any genuine anomaly, and its evidence block (the command's exit code, the regenerated count, `DONE`/`FAILED`).
13. **Rebuild the search index.** When discovery's active-plugin list contains a search-index plugin — the `relevanssi` / `searchwp` families `scripts/classify.py`'s `OPERATIONAL_TABLE_PATTERNS["search_index"]` also tags — probe `ddev wp cli has-command "relevanssi index"` (resp. `ddev wp cli has-command "searchwp index"`); on success run `ddev wp relevanssi index` (resp. `ddev wp searchwp index --rebuild`). A probe failure (free Relevanssi ships no WP-CLI command at all — WP-CLI support is Premium-only, from 1.15.1) is a **report-only fallback**: leave the index empty and surface one clear line in the run report — search index not rebuilt, rebuild manually from wp-admin (Settings → Relevanssi → Build the index). No active search-index plugin → a clean no-op: no probe, no report noise beyond one line. Like thumbnail regeneration, this is local-only, deterministic, and mutates only local index tables, so it runs automatically with no operator gate. **Delegate this phase to `thumbnail-smoke-test`**, folded into the same invocation as the preceding thumbnail-regeneration call; it returns the reindex command's exit code and its outcome (`rebuilt` / `cli-unavailable` / `not-present`) inside its evidence block ([ADR-0015](../../docs/adr/0015-search-index-excluded-and-rebuilt-locally.md)).
14. **Flush with plugins loaded.** `ddev wp rewrite flush --hard` — **without** `--skip-plugins`, so the flush runs with **plugins loaded**. A flush that skips plugins silently drops multilingual and custom routes, 404ing every localised subpage.
15. **Restart.** `ddev restart` to clear the PHP-process caches a `wp cache flush` cannot reach.
16. **Write the new baseline.** Store the locally-filtered current manifest kept in step 4 — path, size, mtime per file, and the **scope** it was taken under — as `.kntnt-wp-skills/last-sync.json` (the `scripts/filter_manifest.py` output over the paged `GET /files` walk), the baseline the next `pull` diffs against.

## 10. Verify (smoke)

Verify from **live state, never assumption**, against a deterministic **expectations** object — never an ad-hoc, hand-checked list. Assemble it from what this run already knows: the discovered core version; the DDEV PHP and database engine/version pins (unchanged at pull, but worth reconfirming); production's table prefix (already verified to match in §4, restated here as the finished-copy expectation); the local DDEV URL (`home`/`siteurl` must equal it, never production's host); entity counts (published posts, pages, users, attachments); the resolved plan's `db_table_content` empty-table list, in full, as `tables.operationalEmpty`, and `tables.contentNonEmpty` restricted to the small set of core tables production is always expected to carry rows for (`posts`, `options`, `users`) — **never** the whole content-table list, since "carried in full" only means the transfer did not silently empty a table, not that production actually put rows in it (`wp_links`, `wp_commentmeta` are full-carried yet legitimately empty on many real sites, and asserting one non-empty would FAIL a correct copy); when step 9.13's reindex rebuilt a search-index plugin's main table, that table too, via `rebuiltSearchIndexTables` — only the table(s) whose rebuild command actually ran, never the whole search-index family, so a genuinely rebuilt index is never asserted empty; the excluded drop-ins expected absent locally — production's discovered `dropins` list **minus** the object-cache drop-in whenever the ownership-rule outcome below resolved to keep it present, since a drop-in can never be expected both absent and present at once; the **object-cache drop-in's expected presence** — the ownership-rule outcome step 9.6 resolved — since a mismatch here means the rule's own verify-and-auto-remove somehow left the wrong state behind; the smoke-test URL list drawn from the copy's own database — the front page plus a couple of real published URLs, and, only if discovery found an active multilingual plugin, the localised home and a real localised subpage, the canary for the rewrite-flush bug; the local-asset check (the front page's URL plus production's bare host); the resolved plan's expected active-plugin count (the preserved inactive set held); confirmation that the saved plan and baseline are expected to exist; and, at pull, that the rollback backup step 9.1 took exists under `.kntnt-wp-skills/backups/`. This is exactly the expectations-file shape `uv run "${CLAUDE_PLUGIN_ROOT}/scripts/smoke_test.py"` consumes — its own module docstring carries the full field list.

**Delegate this phase to `thumbnail-smoke-test`**, but keep this call **separate from, and after,** the preceding thumbnail-regeneration-and-reindex call — `rebuiltSearchIndexTables` above is only correct once step 9.13's reindex outcome (`rebuilt` / `cli-unavailable` / `not-present`) is known, and that outcome exists only inside the *returned* evidence block of that preceding call, so folding this delegation into it would force `rebuiltSearchIndexTables` to be guessed ahead of the reindex, risking a false PASS or FAIL. The one case where combining remains safe: when discovery found no active search-index plugin, step 9.13's reindex is a guaranteed no-op and this call may still fold into the preceding invocation, exactly as before issue #10. Give it the clone directory and the assembled expectations object; it writes the expectations to a scratchpad file and runs `uv run "${CLAUDE_PLUGIN_ROOT}/scripts/smoke_test.py" <clone_dir> <expectations_file>` — the single deterministic check surface that runs every check itself, including the URL fetches asserting the **absence** of `There has been a critical error`, `Fatal error`, and `Error establishing a database` in the HTML, and `ddev wp db check` — and returns only the genuine `fail`/`attention` findings from its JSON report, plus its evidence block: the script's own exit code, its pass/fail/attention/skip summary, and a `DONE`/`FAILED` marker. Filter cosmetic WP-CLI/MariaDB deprecation notices from the report — they are never failures. Re-run `wp db check` yourself as the cheap spot check — an evidence block claiming `DONE` against a nonzero database-check exit code is treated as `FAILED`.

`scripts/smoke_test.py` also runs standalone from a terminal — `uv run "${CLAUDE_PLUGIN_ROOT}/scripts/smoke_test.py" <clone_dir> <expectations.json>` — for a manual re-verification against a hand-edited or previously generated expectations file, and its `--generate` mode snapshots one from a discovery document without hand-writing a baseline.

## 11. Cleanup and report

Remove the large local scratch artifacts (the reassembled dump and the unsealed files). Production is already state-neutral — the extraction job was consumed in step 7 and nothing was left running.

Report the full decided-and-done record. For pull specifically: report the **rollback backup** path (the durable `.kntnt-wp-skills/backups/` file the refresh can be undone from), the trash path if the deletion gate moved anything, the object-cache outcome, and any surfaced define drift. Report the search-index outcome from step 9.13: **rebuilt** (plugin, command, exit 0), **plugin active but CLI unavailable** (manual rebuild instructions surfaced), or **no search-index plugin present**.
