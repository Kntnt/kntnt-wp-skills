# pull

Refresh an existing local DDEV copy of a production WordPress site. `pull` and its sibling `clone` run **one shared transfer engine** — discovery, packing on production, download, verification, remote cleanup, import, and localisation — and differ only at the bookends. A clone is a pull with no baseline; a pull always has one, so only the delta moves: it takes a rollback backup of the local database first, diffs production against the stored baseline so only new and changed files are packed, and re-applies the local state — the inactive plugin set, the object-cache drop-in — afterwards. Every run begins with a **health check** that fails early and cheaply, then walks the operator through a series of **gates** whose recommendations come from the deterministic helpers; `--yes` runs the whole thing unattended and prints a full record. A **risk warning** is always emitted before the destructive local steps.

Read `docs/spec.md` (the specification), `CONTEXT.md` (the glossary — its terms are used verbatim below), and `docs/implementation-notes.md` (the invocation-level literals) alongside this file. Where a literal here and the spec diverge, the spec wins.

## 0. Help gate

If the arguments are `help`, `--help`, or `-h`, run `uv run "${CLAUDE_PLUGIN_ROOT}/scripts/help.py" pull`, emit its output verbatim as Markdown, and stop. Do nothing else — no control-channel call, no file operation.

## How the engine works

**Control channel.** Production is reached **solely** through the Novamira MCP server connected to the live site — never SSH ([ADR-0001](../../docs/adr/0001-novamira-mcp-sole-control-channel.md)). The abilities used are `execute-php`, `run-wp-cli`, `read-file`, `write-file`, and `list-directory`. `run-wp-cli` always takes its arguments as a **JSON array** — a single string silently returns the WP-CLI help text with exit 0. Never deactivate or delete Novamira on production; it is the control channel. `read-file` and `write-file` are **docroot-only**, per Novamira's platform constraints — they cannot see the outside-docroot working dir where the pack artifacts are staged before publication ([ADR-0008](../../docs/adr/0008-encrypted-artifacts-outside-docroot.md)). All outside-docroot IO — fetching `pass.key`, placing the generated `pack.sh` in the working dir — instead goes over `execute-php` with `file_get_contents` / `file_put_contents`, the same authenticated channel. **`pass.key` must never be copied into the docroot, not even transiently** — a docroot copy is web-reachable and defeats the reason it lives outside it.

**The deterministic helper seam.** Every computation that needs neither production nor DDEV is done by a helper script under `${CLAUDE_PLUGIN_ROOT}/scripts/`, invoked with `uv run` and fed JSON on stdin. **Never compute a diff, a classification, a resolved plan, a pack script, or a dump verdict by hand** — the model orchestrates gates and control-channel calls, the helpers decide the shapes ([ADR-0005](../../docs/adr/0005-decision-backbone-gates-and-layered-defaults.md)). The seam is:

- `scripts/discovery.py` — parses the raw health-check and discovery output into the one canonical discovery document.
- `scripts/classify.py` — turns that document into the table split, the define classes, the flagged blobs, the thumbnail exclude-set, and the derived project and directory names.
- `scripts/resolve_plan.py` — resolves the ordered decision list over the layered defaults (`resolve`), and reduces an accepted plan back to the saved plan (`save`).
- `scripts/filter_manifest.py` — restricts production's unfiltered manifest walk to the resolved exclusion scope, locally, so the exclusion set never travels to production (issue #18).
- `scripts/baseline_diff.py` — diffs production's current (locally-filtered) manifest against the stored baseline and returns the new/changed set to pack and the production-deleted set the deletion gate draws from.
- `scripts/pack_script.py` — generates the production-side `pack.sh` from resolved inputs.
- `scripts/dump_sanity.py` — verdicts the decrypted dump against the discovered prefix before the import.

The production-side probes and scans are PHP payloads under `${CLAUDE_PLUGIN_ROOT}/templates/`, sent over `execute-php`; their raw JSON is piped straight into the helpers.

**Run modes.** Interactive walks each gate. `--yes` accepts every recommendation, never pauses, and prints the full decided-and-done record. **Replay** engages automatically when a saved plan exists: interactive collapses to a single *Replay the saved plan?* gate, `--yes` replays silently — except the mass-send valve, which re-surfaces the mail gate even on a silent replay when a freshly-poised campaign would otherwise be masked ([ADR-0009](../../docs/adr/0009-live-mail-default-with-mass-send-valve.md)).

**The gate shape.** Every decision is a **recommendation** behind an accept-or-override gate — *"Recommended: X. Accept? [Y/n]"*. `Y` accepts; `n` reveals the alternatives. Even multi-valued decisions take this shape. The recommendation is whatever `scripts/resolve_plan.py` resolved for that decision; a declined gate records the operator's answer, which the resolver layers above the recommendation.

**Persistent config.** Two per-project files at the local project root: the saved plan `.kntnt-wp-skills.json` (settled per-site answers, committed, all keys optional) and the derived, gitignored `.kntnt-wp-skills/` (baseline manifest `last-sync.json`, rollback backups, trash). The saved plan stores **decisions, never computed lists**, so nothing in it goes stale.

**Subagent delegation.** Four phases — `discovery-classify` (Sonnet, low reasoning effort), `pack-transfer` (Sonnet, medium), `manifest-baseline-diff` (Haiku, low), and `thumbnail-smoke-test` (Haiku, low) — run in pinned subagents defined under `${CLAUDE_PLUGIN_ROOT}/agents/`, so the MCP round-trip logs, curl/checksum output, and thumbnail-regeneration warning spam they produce never enter this agent's own context. Everything else — the health check, every gate, the risk warning, plan resolve, the wp-config edits, and the final report — stays here: subagents run once against a compact JSON task envelope and can never ask the operator anything. Each subagent writes any large payload (raw discovery JSON, the manifest, pack logs) to a scratchpad file and returns only a short summary, the file paths, and its **evidence block** — exit codes, artifact paths with their SHA256, row/file counts, and a `DONE`/`FAILED` marker. **A result missing its evidence block is treated as `FAILED`, whatever its prose claims.** Verification stays deterministic, never a second LLM call: before trusting a subagent's result, re-run the one or two cheap checks its evidence block names — a `sha256sum -c` against a returned checksum, a `wp db check` exit code, or similar.

## 1. Health check

Mandatory step 0 of the engine, before any heavy work. On any failure, abort with a precise remediation message — never a stack trace.

1. **Verify dependencies.** Before any production call, confirm every local and production dependency this run needs — local first, since it is cheaper to fail before ever reaching production. On the first missing dependency, stop immediately: report what to install or fix, where to get it, and the exact command to re-run once done. The agent never installs system software itself; where a fix is safely agent-runnable (e.g. `brew install jq`), offer to run it as its own accept-or-override gate before re-checking, rather than assuming consent. This gate is never auto-accepted under `--yes` — an unattended run has no operator present to consent to installing system software, so `--yes` aborts with the remediation message instead of running the fix.
   - **Local.** `ddev` is on `PATH` and its container backend actually responds — `ddev version` exits zero, and a cheap Docker/Colima liveness probe (`docker info`, else `colima status`) also succeeds; a `ddev` binary with a stopped backend otherwise fails silently deep inside the first `ddev start`. Every required CLI tool is on `PATH`: `uv`, `jq`, `curl`, `shasum` or `sha256sum`, and `openssl`.
   - **Production.** Find the connected Novamira server whose reported home URL matches the target production URL — the operator may have several `novamira-*` servers configured; if several or none match, ask which to use, never guess. Call `discover-abilities` against it and confirm all five abilities this engine relies on are present: `execute-php`, `run-wp-cli`, `read-file`, `write-file`, `list-directory` — the exec probe below (step 4) proves `execute-php` actually works, but only this call inventories the other four before anything downstream assumes them.
2. **Prove the channel is live.** Send `templates/liveness.php` over `execute-php`; it returns the home URL, `ABSPATH`, the PHP version, and the server software. A connected-but-dead channel fails here.
3. **Verify it targets production.** Confirm the returned home URL and root path are production's, **not** the local DDEV site — the verify-targets-prod safety rail. If the channel targets the local site, abort: the production-side steps must never run against the wrong site.
4. **Probe process spawning.** Send `templates/exec-probe.php`: `function_exists('exec')`, the `disable_functions` list, and a live `exec('printf ok')` round-trip. A working `run-wp-cli` does **not** prove this (Novamira may run WP-CLI in-process). If spawning is blocked, abort with the precise cause — the detached pack job would die silently otherwise.
5. **Sweep stranded workspaces.** Send `templates/stranded-sweep.php` to list and remove leftover `kntnt-wp-skills-*` working and download directories from an aborted earlier run (belt-and-braces with the self-destruct timer). Runs before the preflight (step 6) so a batched pair of calls can never delete the preflight's own probe directory; this sweep must never run concurrently with an in-flight preflight.
6. **Preflight the download path.** Send `templates/download-preflight.php`: it writes a tiny **extension-less** test file into a throwaway docroot directory; fetch it over HTTPS from the local side with `curl -fsS`, then delete it. This exercises permissions, extension rules, basic auth, and WAF/CDN behaviour before the multi-gigabyte pack — managed hosts commonly 404 archive extensions.

## 2. Discovery

One read-only production scan. Send `templates/discovery.php` over `execute-php`; it echoes a single JSON object — sizes and versions, the table prefix, the database flavour and collation, InnoDB status, active plugins and any multilingual plugin, the drop-ins and themes, the core version, the mass-send risk scan, the raw attachment metadata, cheap entity counts (published posts, pages, attachments, users), the wp-config defines, and the required-binary probe. The **database password is never returned** — the connection is rebuilt from an allowlist of non-secret constants, so the one secret that unlocks everything never enters model context.

Combine the discovery output with the liveness and exec-probe outputs into one JSON envelope and pipe it to `uv run scripts/discovery.py`; it validates the input and writes the **canonical discovery document** (redacting any secret define value at the boundary). Pipe that document to `uv run scripts/classify.py` to get the classifications: the full-data / empty table split (with form-submission tables tagged their own `user_submissions` category — [ADR-0014](../../docs/adr/0014-user-submissions-own-gate-default-empty.md)), the portable vs auto-excluded defines, the flagged heavy blobs, the thumbnail exclude-set, and the derived project and directory names. If either helper exits non-zero, stop and report its stderr diagnostic — a malformed scan never rides into the run.

**Delegate this phase to `discovery-classify`.** The subagent sends `templates/discovery.php` (with the liveness and exec-probe outputs) over `execute-php`, pipes the envelope through `scripts/discovery.py` and `scripts/classify.py`, writes the canonical discovery document and the classifications to scratchpad files, and returns only their paths, a one-line summary (table, define, and blob counts; the derived project name), and its evidence block: each helper's exit code, the SHA256 and byte size of both written files, the discovered table and active-plugin counts, and a `DONE`/`FAILED` marker. Re-read the written discovery document yourself before resolving the plan against it — a `FAILED` marker or a missing evidence block stops the run here, before any gate is presented.

## 3. Resolve the plan and walk the gates

Assemble the resolve envelope — `{ "operation": "resolve", "skill": "pull", "flags": [...], "discovery": <document>, "classifications": <classifications>, "saved_plan": <.kntnt-wp-skills.json or null>, "answers": {} }` — and run `uv run scripts/resolve_plan.py`. It returns the run mode, whether this is a replay, the ordered **gate** list, and for every decision its **recommendation**, resolved value, and source layer (built-in < live derivation < saved config < this-run answer, with a flag pinning above all four; `--yes` stops at the saved-config layer). The pull decision list carries the bookend decisions clone does not: the preserved inactive plugin set, the object-cache mode, and deletion mirroring. Both skills' decision lists also carry a `user_submissions` gate for form-entry tables the classifier tagged privacy-sensitive — carry/empty, default **empty** ([ADR-0014](../../docs/adr/0014-user-submissions-own-gate-default-empty.md)). Its resolved choice is folded into `db_table_content`'s own resolved value and recommendation — a carry moves the tagged tables from that decision's empty list into its full-data one — so `db_table_content` is always the one list to build the pack step's content/empty table lists from; this is exactly why the walk re-runs `resolve_plan.py` after every answered gate, so a `user_submissions` answer given after `db_table_content` was already walked still folds into the copy that packs.

Walk the returned gates in order. For each, present the recommendation and accept on `Y`; on `n`, reveal the alternatives and record the operator's answer, then re-run `scripts/resolve_plan.py` with that answer in `answers` so the resolved value and every dependent recommendation stay consistent. In `--yes` mode present nothing — every recommendation is accepted — and accumulate the record. When a saved plan exists, present only the single *Replay the saved plan?* gate (plus the mail gate if the valve re-surfaced it).

The mail decision leads with its `findings`: when the mass-send valve flipped, the gate opens with the loud, specific warning naming the engine, the campaign, and the recipient count. Once the walk is done, persist the accepted plan by running `scripts/resolve_plan.py` with `{ "operation": "save", "resolved": <resolved plan>, "saved_plan": <.kntnt-wp-skills.json or null>, "source": { "mcp_server": <the health-checked Novamira server>, "live_url": <discovery `home_url`> } }` and writing the result to `.kntnt-wp-skills.json` — the settled decisions and the persisted `source` (MCP server and live URL), never the computed lists. Pass the prior saved plan back so the save carries forward any key a pull does not walk — notably the DDEV `target` a preceding clone recorded, and the `source` if this run does not resupply it — rather than dropping it on the re-save.

## 4. Pull bookends — verify the prefix, diff the baseline

The pull-only setup over the local site that already exists. Both steps run before the pack: the prefix guard fails cheaply, and the diff decides what the pack even contains.

- **Verify the table prefix.** Read the local table prefix from the marked block in `wp-config.php` and compare it to production's `table_prefix` from discovery. On a **mismatch, abort before any pack or import** — importing production's tables under a prefix the local install does not read leaves WordPress finding zero tables (platform constraint 12). This is the pull manual page's before-import guarantee; because the local prefix already exists and persists, the later marked-block step writes no prefix at pull — it only relies on the match this step already proved.
- **Diff the baseline.** Read the stored baseline `.kntnt-wp-skills/last-sync.json` (path → size + mtime, and the scope it was taken under). Send `templates/manifest.php` — unfiltered, no exclusion payload — over `execute-php` to emit production's **whole** content-tree manifest; a small request, whatever the exclusion set's size (issue #18). Filter it locally: pipe `{ "entries": <the emitted manifest's entries>, "unreadable": <the emitted manifest's unreadable list>, "exclusions": <the resolved exclusion set> }` to `uv run scripts/filter_manifest.py`, which restricts it to the in-scope entries and attaches the resolved set as its scope; it aborts loudly instead if the manifest reports any unreadable directory, since a silently-incomplete tree cannot be trusted for the deletion gate. Combine the stored baseline with that locally-filtered manifest into `{ "baseline": <stored last-sync.json>, "current": <locally-filtered manifest> }` and run `uv run scripts/baseline_diff.py`; it returns `new_or_changed` (the set to pack and pull) and `production_deleted` (the set the deletion gate draws from), the deletion set already restricted to paths still in scope this run (the scope-intersection rule, [ADR-0006](../../docs/adr/0006-baseline-manifest-diff-with-scope.md)). **Never diff the local filesystem** — local mtimes are unreliable through the archive-and-sync chain, so both sides of the diff are production mtimes (platform constraint 19). Keep the locally-filtered current manifest: it becomes this run's new baseline once localisation succeeds.

Only `new_or_changed` is packed and transferred, but **the database is always dumped in full** — trimmed by the empty-table classification it is small and not worth diffing.

**Delegate this phase to `manifest-baseline-diff`.** Fetching production's unfiltered manifest, scope-filtering it locally via `scripts/filter_manifest.py`, and running `scripts/baseline_diff.py` against the stored baseline all run in the `manifest-baseline-diff` subagent: it writes the locally-filtered current manifest and the diff helper's stdout (the `new_or_changed` and `production_deleted` sets) to scratchpad files, and returns their paths, the `new_or_changed` and `production_deleted` counts, and its evidence block: the manifest's row count before and after scope filtering, the helper's exit code, the SHA256 of both written files, and a `DONE`/`FAILED` marker. Read the returned diff-output path for the actual `new_or_changed` and `production_deleted` sets — the pack step below packs the former, the deletion gate itemises the latter. Confirm the manifest's row count structurally against the evidence block's `manifest_scoped_rows` — e.g. `jq '.entries | length' <manifest path>` — before trusting the diff; the manifest is single-line JSON, so `wc -l` on it proves nothing.

## 5. Pack on production (background job)

Generate the production-side pack script — **never assemble this shell by hand**. Build the resolved inputs (working dir, download dir, database name, source root, the **archive set — for pull the `new_or_changed` paths read from the diff-output file the `manifest-baseline-diff` subagent wrote in step 4, not the whole tree** — **an empty exclude list**, the content-table and empty-table lists **from the resolved plan's `db_table_content` decision — never the classifications' raw table split, which `resolve_plan.py` has already folded with the resolved `user_submissions` gate (ADR-0014)**, the InnoDB consistency flag) and pipe them to `uv run scripts/pack_script.py`; it emits `pack.sh`. Unlike clone, pull passes no exclusion list: `new_or_changed` was already diffed against the manifest under the resolved exclusion scope (step 4), so every path it carries is already in scope, and `pack_script.py` skips the exclusion heredoc entirely when the exclude list is empty rather than shipping one that would match nothing.

All packing happens in a working dir **outside the docroot** — `sys_get_temp_dir()`, else a writable dir above `ABSPATH`. If neither is writable, abort rather than fall back to a working dir inside the docroot: `pass.key` is written into that same working dir, and it must never enter the docroot, not even transiently ([ADR-0008](../../docs/adr/0008-encrypted-artifacts-outside-docroot.md)). Over `execute-php` (`file_put_contents`) — the working dir sits outside the docroot, so `write-file` cannot reach it — write into that working dir: the passphrase file `pass.key` (`random_bytes(32)` as hex, mode `0600`), the `.my.cnf` (mode `0600`, built from the DB constants so the **database password never appears on a command line** — consumed via `--defaults-extra-file`), and the generated `pack.sh`. The **passphrase** is passed to OpenSSL by file reference, never as an argument, and is fetched later only over `execute-php` (`file_get_contents`) — never web-served over HTTP.

Launch the detached job: `nohup bash pack.sh >> pack.log 2>&1 & echo $!`. The script dumps the database **in full** in two live-safe passes (`--single-transaction --quick --skip-lock-tables`; full data for the content tables, `--no-data` for the empty-classified ones — a non-InnoDB content table drops `--single-transaction` with a logged caveat), gzips and encrypts to `db.enc`, archives only the new/changed set — already in scope, so no exclusion file applies — straight into `files.enc`, checksums both final names into `SHA256`, publishes the three artifacts world-readable into the docroot download dir, and `touch`es `DONE`. It runs under `set -euo pipefail` with a trap that on failure publishes a `FAILED` marker plus the log tail, and arms a detached self-destruct that removes both directories after a delay even if the client never returns.

**Poll** for the echoed PID: check for `DONE`, `FAILED`, and process liveness (`kill -0 $PID`) up to an explicit maximum wait. On `FAILED`, surface the log tail and abort. A dead process or an exhausted wait aborts with the tail rather than hanging.

## 6. Download, verify, decrypt

Fetch `SHA256` first, then `db.enc` and `files.enc`, each with `curl -fSL -C - --retry 3` into the local scratch area. Verify with `sha256sum -c SHA256` — the artifact names match their creation-time names exactly, never renamed, so the **checksum** verification is honest. A truncated or corrupted transfer is caught here, before anything touches the local site. Then fetch `pass.key` over `execute-php` (`file_get_contents`) — it lives outside the docroot, so `read-file` cannot reach it, and it is never web-served over HTTP — decrypt both artifacts with `openssl enc -d -aes-256-cbc -pbkdf2 -pass file:<local pass.key>`, and `gunzip` the database dump.

## 7. Close the exposure window

Immediately after the checksums pass, delete **both** remote directories over the control channel — the docroot download dir and the outside-docroot working dir, including `pass.key` and `.my.cnf` — and verify they are gone. The self-destruct timer and the next health check's sweep are the backstops; this explicit deletion keeps the exposure window to minutes even so.

**Delegate this phase to `pack-transfer`.** Steps 5–7 — working-dir setup, `pack.sh` generation and launch, polling, download, decryption, and the remote cleanup — run in the `pack-transfer` subagent: it writes the decrypted dump and file archive to scratchpad paths and returns only those paths, the polled log tail, and its evidence block: the pack job's `DONE`/`FAILED` marker and log tail, each artifact's SHA256 as computed on production versus as verified after download, the decrypted-dump and archive byte sizes, and confirmation that both remote directories were deleted. Re-run `sha256sum -c` against the checksums the subagent reports before treating the pack as trustworthy — an evidence block claiming success with a checksum mismatch is treated as `FAILED` regardless.

## 8. Risk warning (always emitted)

Before the destructive local steps, always emit the risk warning itemising the copy's outward-reaching behaviours:

- The resolved mail mode — a live mailer can send real mail.
- A running cron — can fire real webhooks, capture real payments, post to connected social accounts, or re-validate a licence from the dev domain.
- The real user data now in the database.
- **Mandatory:** each per-submission form-to-service integration `classifications.integrations.form_to_service` detected — a form plugin's active service add-on (e.g. WS Form's Mailchimp add-on) fires on a single submit and is invisible to the mass-send valve, which only watches a poised bulk campaign ([ADR-0009](../../docs/adr/0009-live-mail-default-with-mass-send-valve.md)). List every finding's `warning` verbatim, one per bullet — e.g. "submitting form WS Form locally writes to live service Mailchimp" — so submitting a form on the local copy is never mistaken for harmless.

Interactive waits for the operator's confirmation to proceed; `--yes` prints it for the record and proceeds.

## 9. Import and localise

Local and destructive. Run in this exact order — the order is a safety rail ([spec](../../docs/spec.md) *Import and localise*):

1. **Rollback backup (pull only, before anything destructive).** Back up the local database with `ddev export-db` to the durable, gitignored `.kntnt-wp-skills/backups/local-pre-import-<timestamp>.sql.gz` **before any destructive local step** — a bad refresh is then one import away from undone. Keep its path for the report (review point 18 in `docs/implementation-notes.md`).
2. **Sanity-check the dump.** Run `uv run scripts/dump_sanity.py` with the discovered prefix and the resolved plan's `db_table_content` empty-table list (the same folded list step 5 packed with, not the classifications' raw split) against the decrypted dump: the content table created under the prefix, its inserts present, every one of those tables created but empty. Abort on any failure — a wrong-prefix dump leaves WordPress finding nothing.
3. **Import.** `ddev import-db` the decrypted dump; verify table and post counts afterwards.
4. **Extract the files.** Extract `files.enc`'s decrypted archive over the content directory as a **merge**: the local `wp-config.php` and local-only files survive; production's plugin, theme, and media files in the new/changed set overwrite.
5. **Deletions, if enabled (off by default).** Deletion mirroring is **off by default** ([ADR-0010](../../docs/adr/0010-deletion-mirroring-opt-in-trash.md)); when the remembered per-site answer is Yes, itemise the two sources — the `production_deleted` set read from the diff-output file the `manifest-baseline-diff` subagent wrote (step 4) and the plugin/theme **drift** (local plugins/themes with no production counterpart, as a checklist) — and move confirmed items to the timestamped local trash `.kntnt-wp-skills/trash/<timestamp>/`, **never a hard `rm`**, reversible until emptied, and report the path. Under `--yes` nothing is ever removed **without a saved Yes** already on record — a `--yes` run never turns a No, or an unanswered default, into a deletion.
6. **Object-cache drop-in (pull only).** Resolve the drop-in by the **ownership rule** — no local drop-in → nothing; a different owner than production → keep local; the same owner → take production's — then **verify a real request succeeds** and **auto-remove** the object-cache drop-in on failure, reporting it. A production drop-in pointing at a loopback cache host is fatal against DDEV (platform constraint 16).
7. **Write the marked block.** Write the chosen portable defines into the marked block the skills own in `wp-config.php` (the prefix was already verified in step 4 and is not rewritten at pull). **Surface** in the report any **new production define** of the portable class the classifier found — configuration drift brought to attention rather than silently ignored.
8. **Apply mail and cron.** For the resolved mail mode, either keep the existing mailer or install the capture mu-plugin `templates/kntnt-wp-skills-mailpit.php` into `wp-content/mu-plugins` (it short-circuits `wp_mail` at top priority to DDEV's Mailpit, catching API mailers that never touch sendmail). For cron, `--no-cron` writes `define('DISABLE_WP_CRON', true);` into the marked block; otherwise leave it running.
9. **Re-apply the preserved inactive set (pull only).** Re-apply the **preserved inactive** plugin set — derived from live local state this run — with `ddev wp plugin deactivate <list> --skip-plugins --skip-themes`, so plugin and theme code is **skipped during deactivation** and an object-cache plugin cannot re-drop its drop-in mid-step. Tools you keep off stay off.
10. **URL-scoped search-replace.** One pass per source URL form, each `ddev wp search-replace '<old>' '<new>' --all-tables --skip-columns=guid --report-changed-only --skip-plugins --skip-themes`. The forms, in order: `https://www.<domain>`, `http://www.<domain>`, `https://<domain>`, `http://<domain>`, the protocol-relative `//www.<domain>` and `//<domain>`, the **escaped-slash** forms `https:\/\/www.<domain>`, `https:\/\/<domain>`, `http:\/\/www.<domain>`, `http:\/\/<domain>` that page builders store inside JSON, the **escaped-slash protocol-relative** forms `\/\/www.<domain>` and `\/\/<domain>` (a stored protocol-relative URL has no scheme to anchor the scheme-ful escaped passes), the **double-escaped-slash** forms `https:\\/\\/www.<domain>`, `https:\\/\\/<domain>`, `http:\\/\\/www.<domain>`, `http:\\/\\/<domain>` that JSON-within-JSON storage (a plugin's serialised config nested inside another plugin's JSON option) produces — `\\/\\/` is not a substring of `\/\/`, so the single-escaped pass alone leaves it untouched — and, by the same argument, the **double-escaped-slash protocol-relative** forms `\\/\\/www.<domain>` and `\\/\\/<domain>`. **Never the bare domain** — it corrupts `user@domain` email addresses. Serialised objects wp-cli safely skips keep the old domain; harmless.
11. **Set the URLs.** Set the `home` and `siteurl` options explicitly to the DDEV URL.
12. **Regenerate thumbnails (metadata-driven delta, regenerable names only).** `ddev wp media regenerate` scoped to the affected attachment IDs — the **metadata-driven delta** at pull: compare each attachment's registered sizes against the files on disk, but an attachment only joins the regeneration set when a **regenerable-named** size is missing — a name `wp media regenerate` would itself derive from the attachment's *current* attached file, `<stem>-<W>x<H><ext>` (the same rule `scripts/classify.py::thumbnail_exclude_set` applies to the transfer exclusion). An attachment whose only gaps are **non-regenerable** names (production filename drift from a historical WebP conversion, a dedup-suffixed PDF preview, or any other name regeneration can't reproduce) must **never** be fed to `wp media regenerate` — default regenerate deletes-then-rebuilds, so it would delete the transferred drifted file and rebuild it under the wrong (new) name, reintroducing the very 404s the wider transfer scope (#26) was fixed to avoid. This is why change #26 also narrowed the transfer exclusion: after it, a regenerable-named gap should be rare (the file was excluded and needs rebuilding), while a non-regenerable-named gap means the file simply wasn't excluded and already travelled — nothing to regenerate. `--regenerate-all` remains the escape hatch that forces the lot, unfiltered. **Delegate this phase to `thumbnail-smoke-test`**, which runs the delta regeneration, swallows its per-image progress and warning spam, and returns only the regenerated-attachment count, any genuine anomaly, and its evidence block (the command's exit code, the regenerated count, `DONE`/`FAILED`).
13. **Rebuild the search index.** When discovery's active-plugin list contains a search-index plugin — the `relevanssi` / `searchwp` families `scripts/classify.py`'s `OPERATIONAL_TABLE_PATTERNS["search_index"]` also tags — probe `ddev wp cli has-command "relevanssi index"` (resp. `ddev wp cli has-command "searchwp index"`); on success run `ddev wp relevanssi index` (resp. `ddev wp searchwp index --rebuild`). A probe failure (free Relevanssi ships no WP-CLI command at all — WP-CLI support is Premium-only, from 1.15.1) is a **report-only fallback**: leave the index empty and surface one clear line in the run report — search index not rebuilt, rebuild manually from wp-admin (Settings → Relevanssi → Build the index). No active search-index plugin → a clean no-op: no probe, no report noise beyond one line. Like thumbnail regeneration, this is local-only, deterministic, and mutates only local index tables, so it runs automatically with no operator gate. **Delegate this phase to `thumbnail-smoke-test`**, folded into the same invocation as the preceding thumbnail-regeneration call; it returns the reindex command's exit code and its outcome (`rebuilt` / `cli-unavailable` / `not-present`) inside its evidence block ([ADR-0015](../../docs/adr/0015-search-index-excluded-and-rebuilt-locally.md)).
14. **Flush with plugins loaded.** `ddev wp rewrite flush --hard` — **without** `--skip-plugins`, so the flush runs with **plugins loaded**. A flush that skips plugins silently drops multilingual and custom routes, 404ing every localised subpage.
15. **Restart.** `ddev restart` to clear the PHP-process caches a `wp cache flush` cannot reach.
16. **Write the new baseline.** Store the locally-filtered current manifest kept in step 4 — path, size, mtime per file, and the **scope** it was taken under — as `.kntnt-wp-skills/last-sync.json` (the `scripts/filter_manifest.py` output over `templates/manifest.php`'s unfiltered walk), the baseline the next `pull` diffs against.

## 10. Verify (smoke)

Verify from **live state, never assumption**, against a deterministic **expectations** object — never an ad-hoc, hand-checked list. Assemble it from what this run already knows: the discovered core version; the DDEV PHP and database engine/version pins (unchanged at pull, but worth reconfirming); production's table prefix (already verified to match in §4, restated here as the finished-copy expectation); the local DDEV URL (`home`/`siteurl` must equal it, never production's host); entity counts (published posts, pages, users, attachments); the resolved plan's `db_table_content` empty-table list, in full, as `tables.operationalEmpty`, and `tables.contentNonEmpty` restricted to the small set of core tables production is always expected to carry rows for (`posts`, `options`, `users`) — **never** the whole content-table list, since "carried in full" only means the transfer did not silently empty a table, not that production actually put rows in it (`wp_links`, `wp_commentmeta` are full-carried yet legitimately empty on many real sites, and asserting one non-empty would FAIL a correct copy); when step 9.13's reindex rebuilt a search-index plugin's main table, that table too, via `rebuiltSearchIndexTables` — only the table(s) whose rebuild command actually ran, never the whole search-index family, so a genuinely rebuilt index is never asserted empty; the excluded drop-ins expected absent locally — production's discovered `dropins` list **minus** the object-cache drop-in whenever the ownership-rule outcome below resolved to keep it present, since a drop-in can never be expected both absent and present at once; the **object-cache drop-in's expected presence** — the ownership-rule outcome step 9.6 resolved — since a mismatch here means the rule's own verify-and-auto-remove somehow left the wrong state behind; the smoke-test URL list drawn from the copy's own database — the front page plus a couple of real published URLs, and, only if discovery found an active multilingual plugin, the localised home and a real localised subpage, the canary for the rewrite-flush bug; the local-asset check (the front page's URL plus production's bare host); the resolved plan's expected active-plugin count (the preserved inactive set held); confirmation that the saved plan and baseline are expected to exist; and, at pull, that the rollback backup step 9.1 took exists under `.kntnt-wp-skills/backups/`. This is exactly the expectations-file shape `uv run "${CLAUDE_PLUGIN_ROOT}/scripts/smoke_test.py"` consumes — its own module docstring carries the full field list.

**Delegate this phase to `thumbnail-smoke-test`** (it can combine this with the preceding thumbnail-regeneration call in one invocation). Give it the clone directory and the assembled expectations object; it writes the expectations to a scratchpad file and runs `uv run "${CLAUDE_PLUGIN_ROOT}/scripts/smoke_test.py" <clone_dir> <expectations_file>` — the single deterministic check surface that runs every check itself, including the URL fetches asserting the **absence** of `There has been a critical error`, `Fatal error`, and `Error establishing a database` in the HTML, and `ddev wp db check` — and returns only the genuine `fail`/`attention` findings from its JSON report, plus its evidence block: the script's own exit code, its pass/fail/attention/skip summary, and a `DONE`/`FAILED` marker. Filter cosmetic WP-CLI/MariaDB deprecation notices from the report — they are never failures. Re-run `wp db check` yourself as the cheap spot check — an evidence block claiming `DONE` against a nonzero database-check exit code is treated as `FAILED`.

`scripts/smoke_test.py` also runs standalone from a terminal — `uv run "${CLAUDE_PLUGIN_ROOT}/scripts/smoke_test.py" <clone_dir> <expectations.json>` — for a manual re-verification against a hand-edited or previously generated expectations file, and its `--generate` mode snapshots one from a discovery document without hand-writing a baseline.

## 11. Cleanup and report

Remove the large local scratch artifacts (the decrypted dump and archive). Production is already state-neutral — the remote directories were deleted in step 7 and nothing was left running.

Report the full decided-and-done record. For pull specifically: report the **rollback backup** path (the durable `.kntnt-wp-skills/backups/` file the refresh can be undone from), the trash path if the deletion gate moved anything, the object-cache outcome, and any surfaced define drift. Report the search-index outcome from step 13: **rebuilt** (plugin, command, exit 0), **plugin active but CLI unavailable** (manual rebuild instructions surfaced), or **no search-index plugin present**.

This orchestration is a **human-verified residual** ([spec](../../docs/spec.md) *Testing Decisions*): the deterministic helpers it drives are unit-tested at the seam, the engine's own verify phase (step 10) is the in-run verification, and a **manual** end-to-end smoke — a clone followed by a pull against a real site — is the operator's residual before release.
