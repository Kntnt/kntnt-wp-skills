# clone

Create a fresh local DDEV copy of a production WordPress site in an empty directory. `clone` and its sibling `pull` run **one shared transfer engine** — discovery, packing on production, download, verification, remote cleanup, import, and localisation — and differ only at the bookends. A clone is a pull with no baseline: everything is new, so the incremental path is the only path. Every run begins with a **health check** that fails early and cheaply, then walks the operator through a series of **gates** whose recommendations come from the deterministic helpers; `--yes` runs the whole thing unattended and prints a full record. A **risk warning** is always emitted before the destructive local steps.

Read `docs/spec.md` (the specification), `CONTEXT.md` (the glossary — its terms are used verbatim below), and `docs/implementation-notes.md` (the invocation-level literals) alongside this file. Where a literal here and the spec diverge, the spec wins.

## 0. Help gate

If the arguments are `help`, `--help`, or `-h`, run `uv run "${CLAUDE_PLUGIN_ROOT}/scripts/help.py" clone`, emit its output verbatim as Markdown, and stop. Do nothing else — no control-channel call, no file operation.

## How the engine works

**Control channel.** Production is reached **solely** through the Novamira MCP server connected to the live site — never SSH ([ADR-0001](../../docs/adr/0001-novamira-mcp-sole-control-channel.md)). The abilities used are `execute-php`, `run-wp-cli`, `read-file`, `write-file`, and `list-directory`. `run-wp-cli` always takes its arguments as a **JSON array** — a single string silently returns the WP-CLI help text with exit 0. Never deactivate or delete Novamira on production; it is the control channel. `read-file` and `write-file` are **docroot-only**, per Novamira's platform constraints — they cannot see the outside-docroot working dir where the pack artifacts are staged before publication ([ADR-0008](../../docs/adr/0008-encrypted-artifacts-outside-docroot.md)). All outside-docroot IO — fetching `pass.key`, placing the generated `pack.sh` in the working dir — instead goes over `execute-php` with `file_get_contents` / `file_put_contents`, the same authenticated channel. **`pass.key` must never be copied into the docroot, not even transiently** — a docroot copy is web-reachable and defeats the reason it lives outside it.

**The deterministic helper seam.** Every computation that needs neither production nor DDEV is done by a helper script under `${CLAUDE_PLUGIN_ROOT}/scripts/`, invoked with `uv run` and fed JSON on stdin. **Never compute a diff, a classification, a resolved plan, a pack script, or a dump verdict by hand** — the model orchestrates gates and control-channel calls, the helpers decide the shapes ([ADR-0005](../../docs/adr/0005-decision-backbone-gates-and-layered-defaults.md)). The seam is:

- `scripts/discovery.py` — parses the raw health-check and discovery output into the one canonical discovery document.
- `scripts/classify.py` — turns that document into the table split, the define classes, the flagged blobs, the thumbnail exclude-set, and the derived project and directory names.
- `scripts/resolve_plan.py` — resolves the ordered decision list over the layered defaults (`resolve`), and reduces an accepted plan back to the saved plan (`save`).
- `scripts/filter_manifest.py` — restricts production's unfiltered manifest walk to the resolved exclusion scope, locally, so the exclusion set never travels to production (issue #18).
- `scripts/pack_script.py` — generates the production-side `pack.sh` from resolved inputs.
- `scripts/dump_sanity.py` — verdicts the decrypted dump against the discovered prefix before the import.

The production-side probes and scans are PHP payloads under `${CLAUDE_PLUGIN_ROOT}/templates/`, sent over `execute-php`; their raw JSON is piped straight into the helpers.

**Run modes.** Interactive walks each gate. `--yes` accepts every recommendation, never pauses, and prints the full decided-and-done record. **Replay** engages automatically when a saved plan exists: interactive collapses to a single *Replay the saved plan?* gate, `--yes` replays silently — except the mass-send valve, which re-surfaces the mail gate even on a silent replay when a freshly-poised campaign would otherwise be masked ([ADR-0009](../../docs/adr/0009-live-mail-default-with-mass-send-valve.md)).

**The gate shape.** Every decision is a **recommendation** behind an accept-or-override gate — *"Recommended: X. Accept? [Y/n]"*. `Y` accepts; `n` reveals the alternatives. Even multi-valued decisions take this shape. The recommendation is whatever `scripts/resolve_plan.py` resolved for that decision; a declined gate records the operator's answer, which the resolver layers above the recommendation.

**Persistent config.** Two per-project files at the local project root: the saved plan `.kntnt-wp-skills.json` (settled per-site answers, committed, all keys optional) and the derived, gitignored `.kntnt-wp-skills/` (baseline manifest, rollback backups, trash). The saved plan stores **decisions, never computed lists**, so nothing in it goes stale.

**Subagent delegation.** Four phases — `discovery-classify` (Sonnet, low reasoning effort), `pack-transfer` (Sonnet, medium), `manifest-baseline-diff` (Haiku, low), and `thumbnail-smoke-test` (Haiku, low) — run in pinned subagents defined under `${CLAUDE_PLUGIN_ROOT}/agents/`, so the MCP round-trip logs, curl/checksum output, and thumbnail-regeneration warning spam they produce never enter this agent's own context. Everything else — the health check, every gate, the risk warning, plan resolve, the wp-config edits, and the final report — stays here: subagents run once against a compact JSON task envelope and can never ask the operator anything. Each subagent writes any large payload (raw discovery JSON, the manifest, pack logs) to a scratchpad file and returns only a short summary, the file paths, and its **evidence block** — exit codes, artifact paths with their SHA256, row/file counts, and a `DONE`/`FAILED` marker. **A result missing its evidence block is treated as `FAILED`, whatever its prose claims.** Verification stays deterministic, never a second LLM call: before trusting a subagent's result, re-run the one or two cheap checks its evidence block names — a `sha256sum -c` against a returned checksum, a `wp db check` exit code, or similar.

## 1. Health check

Mandatory step 0 of the engine, before any heavy work. On any failure, abort with a precise remediation message — never a stack trace.

1. **Verify dependencies.** Before any production call, confirm every local and production dependency this run needs — local first, since it is cheaper to fail before ever reaching production. On the first missing dependency, stop immediately: report what to install or fix, where to get it, and the exact command to re-run once done. The agent never installs system software itself; where a fix is safely agent-runnable (e.g. `brew install jq`), offer to run it as its own accept-or-override gate before re-checking, rather than assuming consent.
   - **Local.** `ddev` is on `PATH` and its container backend actually responds — `ddev version` exits zero, and a cheap Docker/Colima liveness probe (`docker info`, else `colima status`) also succeeds; a `ddev` binary with a stopped backend otherwise fails silently deep inside the first `ddev start`. `mkwp` is on `PATH` and supports `--dirname` (≥ 1.5.0, [Kntnt/mkwp#2](https://github.com/Kntnt/mkwp/issues/2)) — run `mkwp --help` locally and pipe its captured stdout (`null` if the command is not found at all) to `uv run "${CLAUDE_PLUGIN_ROOT}/scripts/mkwp_guard.py"` as `{ "helpOutput": ... }`; this is the **same** verdict-and-remediation helper the [`mkwp` skill](../mkwp/SKILL.md)'s own version guard reads (its §1) — never re-derived here. Every other required CLI tool is on `PATH`: `uv`, `jq`, `curl`, `shasum` or `sha256sum`, and `openssl`.
   - **Production.** Find the connected Novamira server whose reported home URL matches the target production URL — the operator may have several `novamira-*` servers configured; if several or none match, ask which to use, never guess. Call `discover-abilities` against it and confirm all five abilities this engine relies on are present: `execute-php`, `run-wp-cli`, `read-file`, `write-file`, `list-directory` — the exec probe below (step 4) proves `execute-php` actually works, but only this call inventories the other four before anything downstream assumes them.
2. **Prove the channel is live.** Send `templates/liveness.php` over `execute-php`; it returns the home URL, `ABSPATH`, the PHP version, and the server software. A connected-but-dead channel fails here.
3. **Verify it targets production.** Confirm the returned home URL and root path are production's, **not** the local DDEV site — the verify-targets-prod safety rail. If the channel targets the local site, abort: the production-side steps must never run against the wrong site.
4. **Probe process spawning.** Send `templates/exec-probe.php`: `function_exists('exec')`, the `disable_functions` list, and a live `exec('printf ok')` round-trip. A working `run-wp-cli` does **not** prove this (Novamira may run WP-CLI in-process). If spawning is blocked, abort with the precise cause — the detached pack job would die silently otherwise.
5. **Sweep stranded workspaces.** Send `templates/stranded-sweep.php` to list and remove leftover `kntnt-wp-skills-*` working and download directories from an aborted earlier run (belt-and-braces with the self-destruct timer). Runs before the preflight (step 6) so a batched pair of calls can never delete the preflight's own probe directory; this sweep must never run concurrently with an in-flight preflight.
6. **Preflight the download path.** Send `templates/download-preflight.php`: it writes a tiny **extension-less** test file into a throwaway docroot directory; fetch it over HTTPS from the local side with `curl -fsS`, then delete it. This exercises permissions, extension rules, basic auth, and WAF/CDN behaviour before the multi-gigabyte pack — managed hosts commonly 404 archive extensions.

## 2. Discovery

One read-only production scan. Send `templates/discovery.php` over `execute-php`; it echoes a single JSON object — sizes and versions, the table prefix, the database flavour and collation, InnoDB status, active plugins and any multilingual plugin, the drop-ins and themes, the core version, the mass-send risk scan, the raw attachment metadata, the wp-config defines, and the required-binary probe. The **database password is never returned** — the connection is rebuilt from an allowlist of non-secret constants, so the one secret that unlocks everything never enters model context.

Combine the discovery output with the liveness and exec-probe outputs into one JSON envelope and pipe it to `uv run scripts/discovery.py`; it validates the input and writes the **canonical discovery document** (redacting any secret define value at the boundary). Pipe that document to `uv run scripts/classify.py` to get the classifications: the full-data / empty table split (with form-submission tables tagged their own `user_submissions` category — [ADR-0014](../../docs/adr/0014-user-submissions-own-gate-default-empty.md)), the portable vs auto-excluded defines, the flagged heavy blobs, the thumbnail exclude-set, and the derived project and directory names. If either helper exits non-zero, stop and report its stderr diagnostic — a malformed scan never rides into the run.

**Delegate this phase to `discovery-classify`.** The subagent sends `templates/discovery.php` (with the liveness and exec-probe outputs) over `execute-php`, pipes the envelope through `scripts/discovery.py` and `scripts/classify.py`, writes the canonical discovery document and the classifications to scratchpad files, and returns only their paths, a one-line summary (table, define, and blob counts; the derived project name), and its evidence block: each helper's exit code, the SHA256 and byte size of both written files, the discovered table and active-plugin counts, and a `DONE`/`FAILED` marker. Re-read the written discovery document yourself before resolving the plan against it — a `FAILED` marker or a missing evidence block stops the run here, before any gate is presented.

## 3. Resolve the plan and walk the gates

Assemble the resolve envelope — `{ "operation": "resolve", "skill": "clone", "flags": [...], "discovery": <document>, "classifications": <classifications>, "saved_plan": <.kntnt-wp-skills.json or null>, "answers": {} }` — and run `uv run scripts/resolve_plan.py`. It returns the run mode, whether this is a replay, the ordered **gate** list, and for every decision its **recommendation**, resolved value, and source layer (built-in < live derivation < saved config < this-run answer, with a flag pinning above all four; `--yes` stops at the saved-config layer). Both skills' decision lists carry a `user_submissions` gate for form-entry tables the classifier tagged privacy-sensitive — carry/empty, default **empty** ([ADR-0014](../../docs/adr/0014-user-submissions-own-gate-default-empty.md)). Its resolved choice is folded into `db_table_content`'s own resolved value and recommendation — a carry moves the tagged tables from that decision's empty list into its full-data one — so `db_table_content` is always the one list to build the pack step's content/empty table lists from; this is exactly why the walk re-runs `resolve_plan.py` after every answered gate, so a `user_submissions` answer given after `db_table_content` was already walked still folds into the copy that packs.

Walk the returned gates in order. For each, present the recommendation and accept on `Y`; on `n`, reveal the alternatives and record the operator's answer, then re-run `scripts/resolve_plan.py` with that answer in `answers` so the resolved value and every dependent recommendation stay consistent. In `--yes` mode present nothing — every recommendation is accepted — and accumulate the record. When a saved plan exists, present only the single *Replay the saved plan?* gate (plus the mail gate if the valve re-surfaced it).

The mail decision leads with its `findings`: when the mass-send valve flipped, the gate opens with the loud, specific warning naming the engine, the campaign, and the recipient count. Once the walk is done, persist the accepted plan by running `scripts/resolve_plan.py` with `{ "operation": "save", "resolved": <resolved plan>, "saved_plan": <.kntnt-wp-skills.json or null>, "source": { "mcp_server": <the health-checked Novamira server>, "live_url": <discovery `home_url`> } }` and writing the result to `.kntnt-wp-skills.json` — the settled decisions and the persisted `source` (MCP server and live URL), never the computed lists. Pass the prior saved plan back so a re-save carries forward any key this skill does not walk rather than dropping it (the symmetric guarantee that keeps a pull's later re-save from stripping the DDEV `target` recorded here, or the `source`).

## 4. Clone bookends — name, scaffold, correct the engine, restart

The clone-only setup, using the resolved plan. It runs before the pack because the import needs a live DDEV site to import into. The steps below are strictly ordered — **scaffold, then correct the engine, then restart** — not "pin before any start": `mkwp` (≥ 1.5.0, verified by the dependency step in §1) has no `--db=` flag today, so its own scaffold unconditionally runs `ddev config` on DDEV's default engine, a first `ddev start`, a full placeholder WordPress install, and `ddev launch`, all before control returns here — that first start cannot be deferred past the pin, because `mkwp` performs it internally. A smoke test once let that default-engine database ride unnoticed all the way to import, where a production MariaDB 11.4 dump collided with a scaffolded 11.8, forcing a late reconfigure-and-restart recovery deep inside an already-populated site to undo. The fix below runs that same recovery deliberately, immediately after scaffold, while the project holds nothing but `mkwp`'s own disposable placeholder install — so it costs nothing, rather than a late redo. No step in this bookend, nor anything downstream of it, may import, pack, or otherwise treat this DDEV project as production-shaped before the corrected restart completes.

- **Name-derivation gates.** `scripts/classify.py` derives both names from the production URL — `name`, the DDEV project slug (scheme and `www.` stripped, main label sanitised to the scaffolder's charset) with its DDEV URL, and `directory_name`, the production host verbatim: scheme, userinfo, port, and path stripped, `www.` and every dot kept. `project_name` and `directory_name` are two adjacent decisions in the ordered gate list (each with its own recommendation, saved-plan key, and this-run answer), so the operator accepts or corrects **either independently** through the ordinary accept-or-override gate shape; `--yes` accepts both. Together they cover oddball domains without a public-suffix-list dependency.
- **Scaffold at production's exact core version.** Run `mkwp <name> --dirname=<directory_name> --wp=<core_version>` with the core version from discovery — `mkwp`'s `--dirname` flag (≥ 1.5.0, [Kntnt/mkwp#2](https://github.com/Kntnt/mkwp/issues/2)) names the site directory independently of `NAME`, so the site lands in `<directory_name>` under the operator's current directory while the DDEV project registers as `<name>`. Core files are never transferred. This is the scaffold's own first `ddev start`, on DDEV's default database engine, with a full placeholder WordPress install behind it — nothing from that install is kept. Always quote `<directory_name>` in the command (`--dirname="<directory_name>"`) — `scripts/classify.py` floors it to `FALLBACK_NAME` for shell-metacharacter and leading-hyphen shapes (issue #11), but quoting is the defense-in-depth against an operator-corrected value at the gate.
- **Discard the scaffold's throwaway database.** `ddev stop`, then `ddev delete -Oy` — this drops the default-engine database volume `mkwp` just created (the project's files and `.ddev/config.yaml` survive; the DDEV project registration is recreated by the restart below) so the pin below can take effect. No data loss: nothing but `mkwp`'s own placeholder content ever lived in it.
- **Pin the engine.** Write discovery's `db_engine_php` values into `.ddev/config.yaml`: `ddev config --database=<flavour>:<version> --php-version=<php_major_minor>` (for example `ddev config --database=mariadb:11.4 --php-version=8.3`), so the import does not crash on MySQL-8-vs-MariaDB collations and the copy behaves like production. The database `<version>` is discovery's server version truncated to `major.minor` — DDEV accepts no finer grain, exactly as for PHP — never the raw patch-level string discovery reports. If `mkwp` grows a `--db=` flag, prefer that instead; it does not exist today. Production's table prefix is written into the marked block during localisation (below).
- **Restart on the corrected engine.** `ddev start` brings the project back up against the pinned engine. No step downstream of this restart may run before it completes.

There is no pre-import backup, no preserved inactive set, and no object-cache derivation at clone — nothing local pre-exists.

## 5. Pack on production (background job)

Generate the production-side pack script — **never assemble this shell by hand**. Build the resolved inputs (working dir, download dir, database name, source root, the archive path set, the anchored exclude paths from the classifications, the content-table and empty-table lists **from the resolved plan's `db_table_content` decision — never the classifications' raw table split, which `resolve_plan.py` has already folded with the resolved `user_submissions` gate (ADR-0014)**, the InnoDB consistency flag) and pipe them to `uv run scripts/pack_script.py`; it emits `pack.sh`.

All packing happens in a working dir **outside the docroot** — `sys_get_temp_dir()`, else a writable dir above `ABSPATH`. If neither is writable, abort rather than fall back to a working dir inside the docroot: `pass.key` is written into that same working dir, and it must never enter the docroot, not even transiently ([ADR-0008](../../docs/adr/0008-encrypted-artifacts-outside-docroot.md)). Over `execute-php` (`file_put_contents`) — the working dir sits outside the docroot, so `write-file` cannot reach it — write into that working dir: the passphrase file `pass.key` (`random_bytes(32)` as hex, mode `0600`), the `.my.cnf` (mode `0600`, built from the DB constants so the **database password never appears on a command line** — consumed via `--defaults-extra-file`), and the generated `pack.sh`. The **passphrase** is passed to OpenSSL by file reference, never as an argument, and is fetched later only over `execute-php` (`file_get_contents`) — never web-served over HTTP.

Launch the detached job: `nohup bash pack.sh >> pack.log 2>&1 & echo $!`. The script dumps the database in two live-safe passes (`--single-transaction --quick --skip-lock-tables`; full data for the content tables, `--no-data` for the empty-classified ones — a non-InnoDB content table drops `--single-transaction` with a logged caveat), gzips and encrypts to `db.enc`, archives the in-scope tree through an anchored exclude file (`--anchored --no-wildcards`) straight into `files.enc`, checksums both final names into `SHA256`, publishes the three artifacts world-readable into the docroot download dir, and `touch`es `DONE`. It runs under `set -euo pipefail` with a trap that on failure publishes a `FAILED` marker plus the log tail, and arms a detached self-destruct that removes both directories after a delay even if the client never returns.

**Poll** for the echoed PID: check for `DONE`, `FAILED`, and process liveness (`kill -0 $PID`) up to an explicit maximum wait. On `FAILED`, surface the log tail and abort. A dead process or an exhausted wait aborts with the tail rather than hanging.

## 6. Download, verify, decrypt

Fetch `SHA256` first, then `db.enc` and `files.enc`, each with `curl -fSL -C - --retry 3` into the local scratch area. Verify with `sha256sum -c SHA256` — the artifact names match their creation-time names exactly, never renamed, so the **checksum** verification is honest. A truncated or corrupted transfer is caught here, before anything touches the local site. Then fetch `pass.key` over `execute-php` (`file_get_contents`) — it lives outside the docroot, so `read-file` cannot reach it, and it is never web-served over HTTP — decrypt both artifacts with `openssl enc -d -aes-256-cbc -pbkdf2 -pass file:<local pass.key>`, and `gunzip` the database dump.

## 7. Close the exposure window

Immediately after the checksums pass, delete **both** remote directories over the control channel — the docroot download dir and the outside-docroot working dir, including `pass.key` and `.my.cnf` — and verify they are gone. The self-destruct timer and the next health check's sweep are the backstops; this explicit deletion keeps the exposure window to minutes even so.

**Delegate this phase to `pack-transfer`.** Steps 5–7 — working-dir setup, `pack.sh` generation and launch, polling, download, decryption, and the remote cleanup — run in the `pack-transfer` subagent: it writes the decrypted dump and file archive to scratchpad paths and returns only those paths, the polled log tail, and its evidence block: the pack job's `DONE`/`FAILED` marker and log tail, each artifact's SHA256 as computed on production versus as verified after download, the decrypted-dump and archive byte sizes, and confirmation that both remote directories were deleted. Re-run `sha256sum -c` against the checksums the subagent reports before treating the pack as trustworthy — an evidence block claiming success with a checksum mismatch is treated as `FAILED` regardless.

## 8. Risk warning (always emitted)

Before the destructive local steps, always emit the risk warning itemising the copy's outward-reaching behaviours:

- The resolved mail mode — a live mailer can send real mail.
- A running cron — can fire real webhooks, capture real payments, post to connected social accounts, or re-validate a licence from the dev domain.
- The real user data now in the database.
- **Mandatory:** each per-submission form-to-service integration `classifications.integrations.form_to_service` detected — a form plugin's active service add-on (e.g. WS Form's Mailchimp add-on) fires on a single submit and is invisible to the mass-send valve, which only watches a poised bulk campaign ([ADR-0009](../../docs/adr/0009-live-mail-default-with-mass-send-valve.md)). List every finding's `warning` verbatim, one per bullet — e.g. "submitting form WS Form locally writes to live service Mailchimp" — so submitting a form on the local copy is never mistaken for harmless.

Interactive waits for the operator's confirmation to proceed; `--yes` prints it for the record and proceeds.

## 9. Import and localise

Local and destructive. Run in this exact order — the order is a safety rail ([spec](../../docs/spec.md) *Import and localise*):

1. **Sanity-check the dump.** Run `uv run scripts/dump_sanity.py` with the discovered prefix and the resolved plan's `db_table_content` empty-table list (the same folded list step 5 packed with, not the classifications' raw split) against the decrypted dump: the content table created under the prefix, its inserts present, every one of those tables created but empty. Abort on any failure — a wrong-prefix dump leaves WordPress finding nothing.
2. **Import.** `ddev import-db` the decrypted dump; verify table and post counts afterwards.
3. **Extract the files.** Extract `files.enc`'s decrypted archive over the content directory as a **merge**: the local `wp-config.php` and local-only files survive; production's plugin, theme, and media files overwrite.
4. **Write the marked block.** Write the chosen portable defines and production's table prefix into the marked block the skills own in `wp-config.php`, separate from mkwp's DDEV block.
5. **Apply mail and cron.** For the resolved mail mode, either keep the existing mailer or install the capture mu-plugin `templates/kntnt-wp-skills-mailpit.php` into `wp-content/mu-plugins` (it short-circuits `wp_mail` at top priority to DDEV's Mailpit, catching API mailers that never touch sendmail). For cron, `--no-cron` writes `define('DISABLE_WP_CRON', true);` into the marked block; otherwise leave it running.
6. **URL-scoped search-replace.** One pass per source URL form, each `ddev wp search-replace '<old>' '<new>' --all-tables --skip-columns=guid --report-changed-only --skip-plugins --skip-themes`. The forms, in order: `https://www.<domain>`, `http://www.<domain>`, `https://<domain>`, `http://<domain>`, the protocol-relative `//www.<domain>` and `//<domain>`, and the **escaped-slash** forms `https:\/\/www.<domain>`, `https:\/\/<domain>`, `http:\/\/www.<domain>`, `http:\/\/<domain>` that page builders store inside JSON. **Never the bare domain** — it corrupts `user@domain` email addresses. Serialised objects wp-cli safely skips keep the old domain; harmless.
7. **Set the URLs.** Set the `home` and `siteurl` options explicitly to the DDEV URL.
8. **Regenerate thumbnails.** `ddev wp media regenerate` for every attachment at clone (`--regenerate-all` is the same at clone); the DB-known sizes were excluded from transfer and are rebuilt here. **Delegate this phase to `thumbnail-smoke-test`**, which runs the regeneration, swallows its per-image progress and warning spam, and returns only the regenerated-attachment count, any genuine anomaly, and its evidence block (the command's exit code, the regenerated count, `DONE`/`FAILED`).
9. **Flush with plugins loaded.** `ddev wp rewrite flush --hard` — **without** `--skip-plugins`, so the flush runs with **plugins loaded**. A flush that skips plugins silently drops multilingual and custom routes, 404ing every localised subpage.
10. **Restart.** `ddev restart` to clear the PHP-process caches a `wp cache flush` cannot reach.
11. **Write the baseline.** Send `templates/manifest.php` — unfiltered, no exclusion payload — over `execute-php` to emit production's whole content-tree manifest, then filter it locally: pipe `{ "entries": <the emitted manifest's entries>, "unreadable": <the emitted manifest's unreadable list>, "exclusions": <the resolved exclusion set> }` to `uv run scripts/filter_manifest.py` (issue #18), which restricts it to the in-scope entries — path, size, mtime per file — and attaches the resolved set as its scope; it aborts loudly instead if the manifest reports any unreadable directory, since a silently-incomplete tree cannot be trusted for the deletion gate. Store that locally-filtered manifest as `.kntnt-wp-skills/last-sync.json`, the baseline `pull` diffs against next time. **Delegate this phase to `manifest-baseline-diff`**, run in manifest-only mode — clone has no baseline to diff against, so the subagent fetches production's unfiltered tree, scope-filters it locally via `scripts/filter_manifest.py`, skips `scripts/baseline_diff.py` entirely, and writes the filtered manifest straight to `.kntnt-wp-skills/last-sync.json`; it returns only the write path and its evidence block: the manifest's row count before and after scope filtering, the helper's exit code, the SHA256 of the written file, and a `DONE`/`FAILED` marker.

## 10. Verify (smoke)

Smoke-test from **live state, never assumption**. Build the URL list from the copy's own database: the front page plus a couple of real published URLs; only if discovery found an active multilingual plugin, add the localised home and a real localised subpage — the canary for the rewrite-flush bug. Fetch each and assert a success response and the **absence** of `There has been a critical error`, `Fatal error`, and `Error establishing a database` in the HTML. Run `ddev wp db check`. Confirm the object-cache state and the expected active-plugin count. Filter cosmetic WP-CLI/MariaDB deprecation notices from the report — they are never failures.

**Delegate this phase to `thumbnail-smoke-test`** (it can combine this with the preceding thumbnail-regeneration call in one invocation). The URL fetches, error-string greps, the database check, and the object-cache/active-plugin confirmation all run inside the subagent; it returns only each URL's pass/fail, any anomaly text, and its evidence block: each fetched URL's HTTP status, whether the three error markers were absent, `wp db check`'s exit code, and a `DONE`/`FAILED` marker. Re-run `wp db check` yourself as the cheap spot check — an evidence block claiming `DONE` against a nonzero database-check exit code is treated as `FAILED`.

## 11. Cleanup and report
