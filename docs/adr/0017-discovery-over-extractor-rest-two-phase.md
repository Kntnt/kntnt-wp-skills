# Discovery over the Extractor REST surface: an `environment` endpoint plus a two-phase bootstrap extraction

[ADR-0016](./0016-kntnt-extractor-replaces-novamira-as-control-channel.md) switched the control channel to [Kntnt Extractor](https://github.com/Kntnt/kntnt-extractor) but left one consequence too optimistic: it said the retired `execute-php` discovery logic is "re-expressed as Kntnt Extractor REST calls against its table-list and manifest endpoints." That is not enough. The old single `execute-php` discovery call gathered a batch of **runtime and configuration** facts that Extractor's deliberately generic surface (`status`/`tables`/`files`/`extractions`/`audit-log`) does not expose and cannot: the live PHP version and the database server's flavour, version, and collation (a `mysqldump`-compatible dump carries no server-version header); the home/site URLs; the active-plugins set and whether a multilingual plugin is among them; the `wp-config` defines; the drop-ins present; the per-attachment thumbnail metadata; the cheap entity counts; and the mass-send poised-campaign scan. Some of these are recoverable only by extracting and parsing content; a few have no source at all in files or table dumps; and one class — the defines — is recoverable only in a way that regresses a safety rail. So discovery is a genuine redesign, not a call-site swap. This ADR records that redesign. It refines ADR-0016's discovery consequence; it does not reopen the channel decision.

**Discovery is reconstructed from three sources, in this order.** First, Extractor's **`environment` endpoint** ([kntnt-extractor#15](https://github.com/Kntnt/kntnt-extractor/issues/15)) supplies the runtime/config scalars — PHP version, database server flavour/version/collation, core version, home/site URL, table prefix, content/uploads paths, active plugins, drop-ins — and the **resolved `wp-config` defines with the secret family redacted server-side**. This single authenticated call is also the health check's liveness probe and its verify-targets-production check (via `home_url`), and its success proves the Application Password's user holds both required capabilities. Second, **`GET /tables` and `GET /files`** supply the database size and per-table sizes, the authoritative table enumeration, the uploads size breakdown, the drop-in file presence, the blob-heuristic inputs, and the candidate thumbnail files. Third, a cheap **bootstrap extraction** — a normal `POST /extractions` of `wp_posts`, `wp_postmeta`, the active recognised-mailer tables, and Action Scheduler — is downloaded, unsealed, and parsed **client-side** to compute the three signals that need row-level data: the thumbnail exclude-set (from attachment metadata), the verify-phase entity counts, and the mass-send poised-campaign scan. Only once those are computed is the **main extraction** built and submitted: full-data content tables, **structure-only** empty-classified tables ([kntnt-extractor#16](https://github.com/Kntnt/kntnt-extractor/issues/16)), and the resolved file selection, all sealed to the run's ephemeral X25519 public key. The client unseals the per-segment `KNTNTEXT` container and reassembles it (see *Consequences*).

**Two phases, because classification must precede the heavy pack.** The whole point of the transfer engine is to *not* carry the gigabytes that are excluded or emptied (generated thumbnails, heavy blobs, analytics/log/search-index/form tables — user stories 17, 20, 22). That classification is computed from attachment metadata and table identity, which live in rows, not in the file manifest or the table catalogue. So the signals that shape the main selection have to be extracted and parsed *before* the main selection exists. A single-phase "extract everything, then discard what we did not want" was rejected outright: it ships the very data the classification exists to avoid, defeating user story 17 and bloating every run by potentially many gigabytes. The bootstrap extraction is small (a handful of DB tables, no files) and is consumed before the main one starts, so Extractor's one-active-job concurrency rule is never contended.

**Rejected alternatives, and the safety rails they would have broken.** *Letting the plugin compute the classification or the poised-campaign scan* was rejected because it would push `kntnt-wp-skills` domain knowledge (the regenerable-name thumbnail rule, the blob heuristic, per-engine campaign-table shapes) into a plugin whose entire reason for existing (issue #24) is a minimal, generic surface — the exact "server-side categorisation" Extractor's own ADR-0003 forbids. That boundary holds even though Extractor is in practice this project's companion plugin: the plugin exposes generic primitives, and all domain judgement stays here. *Gating the operator for the PHP and database versions* — the fallback contemplated before the `environment` endpoint existed — is unnecessary now that the endpoint supplies them exactly, and a silently wrong pin would reintroduce the MySQL-8-dump-into-MariaDB collation crash (platform constraint 11) the pin exists to prevent. *Recovering the defines by extracting `wp-config.php` whole* was rejected because it would drag `DB_PASSWORD`, the auth keys, salts, and nonces onto the operator's disk in cleartext, regressing the rule that the database password never leaves production (safety rail 8 / user story 40); the `environment` endpoint resolves the non-secret defines and redacts the secret family server-side, so the secrets are never fetched at all — the rail is preserved, arguably strengthened.

## Consequences

- The health check's production step becomes: `GET /status` (the API-version handshake — the run refuses an Extractor whose API version is below the floor these skills require) followed by `GET /environment` on the configured Application Password. The `environment` call is the liveness proof, the verify-targets-production check (`home_url` must match the intended production URL and must not be the local DDEV site), and the capability proof in one round trip — a `200` shows the user holds both `kntnt_extractor_operate` and `manage_options`; a `403` fails fast with a precise, per-capability remediation (disambiguated, if useful, by whether `GET /audit-log`, which is `manage_options`-only, also refuses). No Novamira ability inventory, `execute-php` exec-probe, or file-ability probe remains.
- The six `templates/*.php` `execute-php` payload bodies are retired, as ADR-0016 already required, and their logic re-homes precisely: `discovery.php`'s runtime/config half → the `environment` endpoint; its domain half (thumbnail metadata, mass-send scan, entity counts) → client-side parsing of the bootstrap extraction; `manifest.php` → `GET /files`; `liveness.php` and `exec-probe.php` → obsolete (the `environment` call proves liveness, and process-spawning no longer matters because packing runs inside the plugin, not as a detached job these skills launch); `stranded-sweep.php` → `GET /extractions` plus `DELETE /extractions/{id}` ([kntnt-extractor#17](https://github.com/Kntnt/kntnt-extractor/issues/17)), which lets the health check cancel a job a crashed earlier run stranded; `download-preflight.php` → a throwaway minimal extraction round-tripped through its one-time `download_url`, which exercises the real serving path before the heavy pack.
- The client gains a sealed-container unseal helper in the deterministic helper seam — a standalone Python script run via `uv` with `pynacl` as an inline dependency — that generates the run's ephemeral X25519 key pair, unseals each segment's key with the private key (`crypto_box_seal` open), decrypts each segment (`crypto_secretbox` open), and reassembles the container: table segments concatenated into one importable `.sql`, file segments written to disk by their installation-root-relative path. The plugin's per-table DDL is `mysqldump`-compatible but carries no global preamble, so the helper prepends the connection-safe preamble (charset `SET NAMES`, `FOREIGN_KEY_CHECKS`, SQL mode) the reassembled dump needs to import cleanly into DDEV.
- The DDEV database-engine and PHP pins (user story 19, platform constraint 11) are driven directly by the `environment` endpoint's exact values — no operator gate, no inference.
- The production-side pack machinery these skills used to generate and run — `pack.sh`, the server-side passphrase, the `openssl` encryption, the detached background job, the self-destruct timer, the docroot download dir (ADR-0007, ADR-0008 as *implemented here*) — is retired in full. Its intent is preserved, but now lives inside Extractor: the plugin owns the background extraction job, the per-segment sealing, the one-time download link, and the TTL/watchdog cleanup. `kntnt-wp-skills` no longer generates a shell script, never handles a passphrase, and no longer needs `scripts/pack_script.py`. ADR-0007 and ADR-0008 stay as the historical record of how the exposure model was met before the cutover; they are not rewritten.
- These skills require Extractor at an API version that includes the `environment` endpoint, structure-only extraction, and caller job listing (the three changes ship together as API version 2). The `status` handshake enforces that floor and refuses an older install with a precise upgrade instruction.
