# Kntnt Extractor replaces Novamira as the control channel — a hard cutover, superseding ADR-0001

`clone`/`pull` now reach production solely through [Kntnt Extractor](https://github.com/Kntnt/kntnt-extractor), a purpose-built companion plugin, instead of the Novamira MCP server ([issue #24](https://github.com/Kntnt/kntnt-wp-skills/issues/24)). This is a **hard cutover**, not a transitional period supporting both: `kntnt-wp-skills` is still early (one release out), Novamira's only foothold is the `mkwp --plugins` recommendation, and carrying two control-channel implementations side by side would be complexity with no lasting payoff given the operator's intent to fully retire Novamira. [ADR-0001](./0001-novamira-mcp-sole-control-channel.md) is superseded in its entirety by this decision; its text is left as the historical record and is not rewritten.

The substance of the change is a large reduction in attack surface: Novamira's `execute-php`/`run-wp-cli` abilities allow arbitrary code execution on production, far broader than the transfer engine ever needed. Kntnt Extractor exposes only a fixed, minimal REST surface — list tables, list files, extract an explicit selection, poll a job, fetch a one-time download link, read the audit log — authenticated as a real WordPress user whose own capabilities bound what they can extract. The plugin's own authentication, authorization, extraction-job, delivery, distribution, and audit-log design is out of scope here and lives in that project's own `CONTEXT.md`/`docs/adr/`; this ADR covers only what changes for `kntnt-wp-skills` itself.

## Consequences

- The health check's dependency step no longer probes Novamira's `discover-abilities`, the `execute-php` exec-probe, or `read-file`/`write-file`/`list-directory`; it instead calls Kntnt Extractor's unauthenticated `status` endpoint (API-version handshake) and verifies the configured Application Password's user holds both `kntnt_extractor_operate` and `manage_options`. This corrects the authorization model this ADR originally anticipated: Kntnt Extractor as built ([its ADR-0002](https://github.com/Kntnt/kntnt-extractor/blob/main/docs/adr/0002-authn-app-password-authz-operate-plus-manage-options.md)) gates *every* listing and extraction on a flat `manage_options`, not on a per-table/file mapping to whatever capability governs that resource — so there is no per-resource capability set for the health check to compute or verify. `kntnt_extractor_operate` is the "on switch" and `manage_options` authorises the data; a user with one but not the other reaches nothing.
- `templates/*.php` (the `execute-php` payload bodies for discovery, manifest, liveness, exec-probe, stranded-sweep, download-preflight) are retired; their read-only logic is re-expressed as Kntnt Extractor REST calls against its table-list and manifest endpoints.
- `mkwp --plugins`' Novamira-by-default recommendation switches to Kntnt Extractor, exactly as anticipated in `docs/spec.md`'s existing note on this issue.
- The artifact exposure model (packing outside the docroot, encrypted `.enc` artifacts, a briefly-published download path) is preserved in substance — Kntnt Extractor's own background-job-plus-one-time-link design fills the same role Novamira's `execute-php`-driven pack script did; `kntnt-wp-skills` no longer needs its own `pack.sh` template or the `execute-php`/`read-file` passphrase dance, since that mechanics now lives entirely inside the plugin.
- Novamira is no longer installed, recommended, or referenced by any run; nothing in `kntnt-wp-skills` deactivates or removes an existing Novamira installation, but new runs never touch it.
