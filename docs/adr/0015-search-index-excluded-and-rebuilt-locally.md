# Search index excluded from transfer, rebuilt locally via the plugin's own WP-CLI command

`scripts/classify.py`'s `OPERATIONAL_TABLE_PATTERNS["search_index"]` carries Relevanssi's and SearchWP's index tables empty (schema-only) at both clone and pull, the same as the other three operational families (analytics, cookie-consent, email-log) — the dump stays small (spec user story 17) and nothing genuinely irreplaceable is lost, because a search index is entirely derived from content already in the database. But unlike those three, which are discard-and-forget logs nobody needs back, a search index is **derived functional data** the copy is expected to actually work with: local search returns nothing until something rebuilds it. [ADR-0011](./0011-metadata-driven-thumbnail-regeneration.md) already draws exactly this line for generated thumbnails — excluded from transfer because they are large and fully derivable, then regenerated locally after import — and this ADR extends the same reasoning to the search index: exclude it from transfer as before, but add a local rebuild step immediately after import, folded into the same `thumbnail-smoke-test` subagent invocation that already regenerates thumbnails (issue #13).

## Decision

The reindex runs automatically after import, with **no operator gate** — it is local-only, deterministic, mutates only local index tables, and restores fidelity the transfer deliberately dropped, the same footing ADR-0011's thumbnail regeneration already stands on. Whether the plugin's WP-CLI command exists is **capability-probed, never assumed**: `wp cli has-command "relevanssi index"` / `wp cli has-command "searchwp index"` (exit 0 = available) distinguishes Relevanssi Premium (which ships the command from 1.15.1) from free Relevanssi (which has none) without guessing at plugin slugs or version strings. On success, the family's documented rebuild command runs — `wp relevanssi index` (Relevanssi's own manual calls it "the fastest and easiest way to rebuild the index") or `wp searchwp index --rebuild`. On probe failure, the fallback is **report-only**: leave the index empty and surface a single clear line in the run report naming the manual path (wp-admin → Settings → Relevanssi → Build the index) — never a `wp eval` workaround or any other undocumented internal reaching into the plugin's own indexing internals. `DISABLE_WP_CRON` and the local cron choice are irrelevant here: the rebuild is a direct CLI invocation the orchestration runs itself, not a cron-driven background index, so it behaves identically whichever cron choice the plan resolved.

Command facts verified against the vendor docs (2026-07-20): [Relevanssi's WP-CLI manual](https://www.relevanssi.com/user-manual/wp-cli/) and [SearchWP's WP-CLI docs](https://searchwp.com/documentation/wp-cli/).

## Consequences

- `scripts/smoke_test.py`'s `generate_expectations` gains the `rebuiltSearchIndexTables` override (issue #10): the orchestration passes only the main index table(s) of a plugin whose rebuild command actually ran, and each named table moves from `tables.operationalEmpty` into `tables.contentNonEmpty` — a copy that genuinely rebuilt its index must never fail its own smoke test for having a non-empty table the classifier's raw split still expects empty. The log/stopwords/tracking siblings a plugin like Relevanssi ships alongside its main table legitimately stay empty; only the main table is asserted non-empty.
- The classifier's own table split (`scripts/classify.py`) is unchanged — `search_index` stays folded into the same silent empty-by-default family as the other three operational categories. This ADR only adds a local rebuild after import; what travels in the dump is untouched.
- Out of scope, deliberately: transferring production's own index (defeats the point — it is large and fully derivable), `wp relevanssi refresh` (the non-clearing variant; this rebuild always starts from empty, so the clearing `index` command is correct), `--target=users|taxonomies` partial passes, and any search plugin outside the classifier's `relevanssi`/`searchwp` families.

Settled at triage from issue #10, surfaced while preparing the real-site smoke test against smoltek.com (Relevanssi-backed).
