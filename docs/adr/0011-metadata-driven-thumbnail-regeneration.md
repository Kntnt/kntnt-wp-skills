# Exclude only DB-known thumbnails; regenerate metadata-driven deltas, not file diffs

We can only safely exclude a thumbnail from transfer if something will rebuild it, and only `wp media regenerate` can — and it operates strictly on **DB-registered attachments** (verified: it cannot process images with no attachment record). So the exclude set is exactly the DB-known generated sizes (`_wp_attachment_metadata → sizes[*].file`); everything else — registered originals and side-loaded/orphan files, including their thumbnails — is pulled whole because it cannot be regenerated. The DB-derived set also resolves ambiguous names (a `banner-1920x1080.jpg` original is `_wp_attached_file`, kept; its derivatives are in `sizes[*].file`, dropped) with no filename heuristic.

At `pull`, regeneration is **metadata-driven, not file-diff-driven**: every `pull` re-imports the full content DB, so newly-registered sizes are already local right after import, and the engine compares each attachment's registered `sizes[*]` against the files on disk and regenerates exactly the deltas. This catches the case a file diff misses — production registered a *new* size (e.g. after a theme change) without the original changing. Because default `wp media regenerate` deletes-then-rebuilds, a changed original's stale thumbnails are wiped and rebuilt when its ID regenerates. `--regenerate-all` is the sledgehammer escape hatch.

## Amendment (2026-07-20): "DB-known ⇒ regenerable" does not hold on real sites; exclusion and the regeneration delta both narrow to regenerable-named sizes

Investigation of #21, against a real production site (smoltek.com), found this ADR's founding premise false for a material share of its media library: **260 of 658 attachments (2,648 of 6,129 registered size entries)** carried `_wp_attachment_metadata → sizes[*].file` names that `wp media regenerate` can never reproduce, because they were written by a historical event outside WordPress's own naming — a bulk WebP conversion (`…mv2-300x211.webp`, derived from the *pre-conversion* original, when the current attached file is `…mv2-jpg.webp` and regeneration derives `…mv2-jpg-300x211.webp`) and dedup-suffixed PDF previews (`…-pdf-2.webp`, a collision suffix regeneration never inserts). One further attachment (a favicon) carried a canonical-shaped registered size WordPress no longer produces at all.

The consequence was two-fold and self-reinforcing. First, the exclude-set (as originally defined — every DB-known size, full stop) dropped these exact names from transfer, and nothing ever rebuilt them under their production names, so hardcoded `<img>`/`srcset` references to them 404 locally forever — the **inverse** of this ADR's own safety rule ("we can only safely exclude a thumbnail if something will rebuild it"). Second, at `pull`, every re-import of production metadata re-registers the same drifted names, so the metadata-driven delta re-flags the identical 260 attachments on every run — a permanent phantom residue, measured stable across repeated replays (#21's diagnosis), not transient noise.

**Decision:** narrow both the transfer exclusion and the pull-side regeneration delta to sizes whose name is **regenerable** — exactly what `wp media regenerate` would itself derive from the attachment's *current* attached file: `<stem>-<W>x<H><ext>`, where `<stem>` and `<ext>` come from the `file` basename and `<W>x<H>` is a literal, case-sensitive dimensions token terminal before the extension. Concretely:

- `scripts/classify.py::thumbnail_exclude_set` excludes a registered size from transfer only when its name matches that derivation; a non-matching name (conversion drift, a dedup suffix, an extension change) stays **in** the transfer, exactly like any other file nothing can regenerate (a side-loaded thumbnail, an orphan).
- The pull-side metadata-driven delta applies the identical rule: an attachment joins the `wp media regenerate` scope only when a *regenerable-named* size is missing on disk. An attachment whose only gaps are non-regenerable names is never sent to regenerate — default regenerate deletes-then-rebuilds, so feeding it a transferred, correctly-drifted-named file would delete that file and rebuild it under the wrong (new) name, reintroducing the exact 404s this amendment exists to stop.

This restores the ADR's own premise rather than abandoning it: "DB-known ⇒ excludable" becomes "DB-known **and regenerable-named** ⇒ excludable," and everything DB-known but not regenerable-named now falls into the ADR's own existing "cannot be regenerated, so it is pulled whole" branch — the same branch side-loaded and orphan files were already in. No new branch, no new default; the classification the ADR always drew is simply applied correctly. A residual is accepted and documented rather than chased further: a canonical-shaped name WordPress no longer actually produces for that attachment (the favicon case) is indistinguishable from a genuinely regenerable one by name alone, since the classifier only ever sees names, never runs `wp media regenerate` itself to know what it would produce; this is bounded, known (four attachments on the evidence site), and immaterial next to the 260-attachment residue the amendment removes.
